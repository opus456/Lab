# 15 综合实战项目：类型安全的 CLI 配置检查工具

> **难度：实战** | 本章把 TypeScript 的类型系统、模块化、工程配置、声明文件、Node CLI 和测试串成一个真实可用项目。

---

## 1. 项目目标

**标注：核心高频**

构建一个 `config-guard` CLI，用来检查项目中的 JSON 配置文件是否符合团队约定。

```bash
config-guard check ./app.config.json
config-guard check ./app.config.json --schema ./schema.json
config-guard init
```

核心能力：

- 校验配置字段类型
- 输出清晰错误路径
- 支持默认 schema
- 支持 JSON 报告输出
- 支持单元测试和 CLI 集成测试

**为什么这样设计、这样设计带来的好处：**

配置错误是工程项目高频问题。把校验工具做成 CLI，可以接入本地开发、CI 和 pre-commit 流程。

## 2. 技术选型

**标注：重要**

| 需求 | 推荐方案 | 理由 |
|------|----------|------|
| CLI 参数 | `commander` | 成熟、简单、生态稳定 |
| 运行 TS | `tsx` | 开发阶段零配置运行 TypeScript |
| 构建 | `tsup` | 输出 ESM/CJS 简单高效 |
| Schema 校验 | `zod` | 运行时校验与静态类型联动 |
| 测试 | `vitest` | 快速、类型友好 |

```json
{
  "scripts": {
    "dev": "tsx src/cli.ts",
    "build": "tsup src/cli.ts --format esm,cjs --dts",
    "test": "vitest"
  }
}
```

**为什么这样设计、这样设计带来的好处：**

TypeScript 类型只在编译期存在，配置文件来自运行时，必须用 `zod` 这类运行时校验库补上边界安全。

## 4. 核心类型设计

**标注：核心高频**

```typescript
import { z } from "zod";

export const AppConfigSchema = z.object({
  appName: z.string().min(1),
  port: z.number().int().min(1).max(65535),
  env: z.enum(["development", "test", "production"]),
  features: z.record(z.boolean()).default({}),
});

export type AppConfig = z.infer<typeof AppConfigSchema>;
```

`z.infer` 可以从运行时 schema 推导静态类型，避免“类型定义一份、校验规则一份”导致漂移。

**为什么这样设计、这样设计带来的好处：**

TypeScript 无法直接验证 JSON 文件。用 schema 作为单一事实来源，可以同时服务运行时校验和编译期类型。

## 5. 校验流程

**标注：核心高频**

```typescript
import { readFile } from "node:fs/promises";
import { AppConfigSchema } from "./schema";

export async function checkConfig(path: string) {
  const raw = await readFile(path, "utf-8");
  const json = JSON.parse(raw) as unknown;
  return AppConfigSchema.safeParse(json);
}
```

流程图：

```text
读取文件 -> JSON.parse -> unknown -> schema.safeParse -> 成功配置 / 结构化错误
```

**为什么这样设计、这样设计带来的好处：**

从外部读取的数据应先视为 `unknown`，不能直接断言为业务类型。这样能防止假类型安全。

## 6. CLI 入口

**标注：重要**

```typescript
import { Command } from "commander";
import { checkConfig } from "./checker";

const program = new Command();

program
  .command("check")
  .argument("<path>")
  .option("--json", "output JSON report")
  .action(async (path, options) => {
    const result = await checkConfig(path);
    if (!result.success) {
      console.error(result.error.format());
      process.exitCode = 1;
      return;
    }
    console.log(options.json ? JSON.stringify(result.data) : "Config OK");
  });

program.parseAsync();
```

**为什么这样设计、这样设计带来的好处：**

CLI 不应该吞掉错误，也不应该到处 `process.exit()`。设置 `process.exitCode` 更利于测试和资源清理。

## 7. 测试策略

**标注：重要**

- **单元测试**：校验 schema 和错误路径
- **集成测试**：执行 CLI，检查退出码和输出
- **类型测试**：确保导出的类型符合预期

```typescript
import { describe, expect, it } from "vitest";
import { AppConfigSchema } from "../src/schema";

describe("AppConfigSchema", () => {
  it("accepts valid config", () => {
    const result = AppConfigSchema.safeParse({
      appName: "demo",
      port: 3000,
      env: "development",
    });

    expect(result.success).toBe(true);
  });
});
```

**为什么这样设计、这样设计带来的好处：**

类型系统不能替代测试。测试验证运行时行为，类型系统验证开发期约束，两者互补。
