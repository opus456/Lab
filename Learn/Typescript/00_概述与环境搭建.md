# TypeScript 概述与环境搭建

## 1. TypeScript 是什么

### 历史背景

TypeScript 由微软于 2012 年发布，由 C# 之父 Anders Hejlsberg 主导设计。当时 JavaScript 正在从浏览器脚本语言向大规模应用开发语言转变，但它缺乏静态类型系统，导致大型项目难以维护。

**微软为什么创造 TypeScript？**

2010 年前后，微软内部多个大型项目（如 Bing Maps、Office Online）使用 JavaScript 开发，团队遇到了严重的工程化问题：

- 代码重构极其困难，改一个函数签名不知道会影响哪些调用方
- IDE 无法提供可靠的自动补全和跳转
- 新人接手项目时，只能靠注释和文档理解接口约定
- 运行时才能发现的类型错误，在生产环境中造成事故

```javascript
// 一个真实场景：某电商系统的订单处理
function processOrder(order) {
  // order 是什么结构？有哪些字段？哪些是必填的？
  // 三个月后回来看这段代码，完全不知道
  const total = order.items.reduce((sum, item) => sum + item.price * item.qty, 0);
  return { ...order, total, status: 'processed' };
}
```

**TypeScript 解决了什么问题？**

TypeScript 是 JavaScript 的超集（superset），它在 JavaScript 之上添加了可选的静态类型系统。核心价值：

1. **编译时错误检查** - 在代码运行之前就发现问题
2. **代码即文档** - 类型注解本身就是最好的文档
3. **IDE 智能支持** - 精确的自动补全、重构、跳转
4. **渐进式采用** - 可以逐步将 JS 项目迁移到 TS

```typescript
// 同样的代码，TypeScript 版本
interface OrderItem {
  name: string;
  price: number;
  qty: number;
}

interface Order {
  id: string;
  items: OrderItem[];
  customer: string;
}

function processOrder(order: Order): Order & { total: number; status: string } {
  const total = order.items.reduce((sum, item) => sum + item.price * item.qty, 0);
  return { ...order, total, status: 'processed' };
}
```

现在任何人看到这段代码，都能立刻知道 `order` 的结构，IDE 也能提供精确的补全。
## 2. TypeScript vs JavaScript - 详细对比

### 为什么需要类型系统

先看几个实际工程中因为缺乏类型而产生的 bug：

**Bug 1：隐式类型转换**

```javascript
// JavaScript - 线上真实事故
function calculateDiscount(price, discount) {
  return price - price * discount;
}

// 从表单获取的值是字符串，但没人注意到
const finalPrice = calculateDiscount("100", 0.2);
// 结果："100" - "100" * 0.2 = "100" - 20 = 80 碰巧对了

const finalPrice2 = calculateDiscount("100", "20%");
// 结果：NaN，线上用户看到价格显示 NaN
```

```typescript
// TypeScript - 编译时就会报错
function calculateDiscount(price: number, discount: number): number {
  return price - price * discount;
}

calculateDiscount("100", 0.2);
// Error: Argument of type 'string' is not assignable to parameter of type 'number'
```

**Bug 2：属性名拼写错误**

```javascript
// JavaScript - 静默失败
const user = { name: "Alice", email: "alice@example.com" };
console.log(user.emial); // undefined，不会报错，但逻辑错了
```

```typescript
// TypeScript - 立即报错
interface User {
  name: string;
  email: string;
}
const user: User = { name: "Alice", email: "alice@example.com" };
console.log(user.emial);
// Error: Property 'emial' does not exist on type 'User'. Did you mean 'email'?
```

**Bug 3：函数参数顺序错误**

```javascript
// JavaScript - 参数传反了，运行时才发现
function createUser(name, age, email) {
  return { name, age, email };
}
// 新同事不熟悉接口，把参数顺序搞反了
const user = createUser("Alice", "alice@example.com", 25);
// { name: "Alice", age: "alice@example.com", email: 25 } 完全错误
```
```typescript
// TypeScript - 编译器帮你检查
function createUser(name: string, age: number, email: string) {
  return { name, age, email };
}
createUser("Alice", "alice@example.com", 25);
// Error: Argument of type 'string' is not assignable to parameter of type 'number'
```

### 核心差异对比表

| 特性 | JavaScript | TypeScript |
|------|-----------|------------|
| 类型系统 | 动态类型 | 静态类型（可选） |
| 错误发现时机 | 运行时 | 编译时 |
| IDE 支持 | 有限 | 完整（补全、重构、跳转） |
| 学习曲线 | 低 | 中等 |
| 代码可读性 | 依赖注释 | 类型即文档 |
| 运行环境 | 直接运行 | 需要编译为 JS |
| 生态兼容性 | 原生 | 通过 .d.ts 声明文件兼容 |

### 什么时候用 TypeScript？

- **推荐使用**：团队协作项目、长期维护的项目、库/框架开发、中大型应用
- **可以不用**：一次性脚本、快速原型验证、极小的工具函数

**为什么这样设计？** TypeScript 选择"可选类型"而非"强制类型"，是为了降低迁移成本。
任何合法的 JavaScript 代码都是合法的 TypeScript 代码，这意味着团队可以渐进式地
为现有项目添加类型，而不需要一次性重写。
## 3. TypeScript 的设计哲学

### 结构化类型系统（Structural Type System）

TypeScript 采用结构化类型系统，也叫"鸭子类型"（Duck Typing）的静态版本。
判断两个类型是否兼容，看的是**结构**（有哪些属性和方法），而不是**名字**。

```typescript
interface Point {
  x: number;
  y: number;
}

interface Coordinate {
  x: number;
  y: number;
}

const p: Point = { x: 1, y: 2 };
const c: Coordinate = p; // 完全合法！结构相同就兼容
```

### 名义类型系统（Nominal Type System）

对比 Java/C# 的名义类型系统：

```java
// Java - 名义类型系统
class Point { int x; int y; }
class Coordinate { int x; int y; }

Point p = new Point();
Coordinate c = p; // 编译错误！即使结构完全相同，名字不同就不兼容
```

### 为什么 TypeScript 选择结构化类型？

1. **与 JavaScript 运行时行为一致** - JS 本身就是鸭子类型，TS 的类型系统是对 JS 运行时行为的静态建模
2. **更灵活的代码组合** - 不需要显式声明"实现了某个接口"
3. **更好的第三方库兼容性** - 不同库定义的相似类型可以互相赋值

```typescript
// 结构化类型的实际好处：不同库的类型可以无缝协作
// 假设库 A 定义了：
interface Serializable {
  toJSON(): string;
}

// 你的代码中任何有 toJSON 方法的对象都自动满足这个接口
class MyData {
  constructor(private value: number) {}
  toJSON(): string {
    return JSON.stringify({ value: this.value });
  }
}

// 不需要 class MyData implements Serializable
// 结构匹配就够了
function save(data: Serializable) {
  console.log(data.toJSON());
}
save(new MyData(42)); // 合法，因为 MyData 的结构满足 Serializable
```
### 模拟名义类型（Branded Types）

有时候结构化类型太宽松了，比如你不想让"用户ID"和"订单ID"互相赋值：

```typescript
// 使用 brand 技术模拟名义类型
type UserId = string & { readonly __brand: unique symbol };
type OrderId = string & { readonly __brand: unique symbol };

function createUserId(id: string): UserId {
  return id as UserId;
}

function createOrderId(id: string): OrderId {
  return id as OrderId;
}

function getUser(id: UserId) { /* ... */ }

const userId = createUserId("user_123");
const orderId = createOrderId("order_456");

getUser(userId);  // 合法
getUser(orderId); // Error: 类型不兼容
```

**设计哲学总结：** TypeScript 的类型系统不是为了限制程序员，而是为了精确描述
JavaScript 的运行时行为，在不改变运行时语义的前提下提供静态检查。
## 4. 环境搭建

### 4.1 Node.js 安装

TypeScript 编译器运行在 Node.js 上。推荐使用 LTS（长期支持）版本。

**安装方式选择：**

| 方式 | 适用场景 | 说明 |
|------|---------|------|
| 官网下载 | 初学者 | https://nodejs.org 下载 LTS 版本 |
| nvm (Linux/Mac) | 需要多版本切换 | `nvm install --lts` |
| nvm-windows | Windows 多版本 | `nvm install lts` |
| fnm | 跨平台，更快 | `fnm install --lts` |

```bash
# 验证安装
node --version   # 应该显示 v20.x 或更高
npm --version    # 应该显示 10.x 或更高
```

### 4.2 包管理器选择：npm vs pnpm

| 特性 | npm | pnpm |
|------|-----|------|
| 磁盘占用 | 每个项目独立副本 | 全局存储 + 硬链接，节省大量空间 |
| 安装速度 | 较慢 | 快 2-3 倍 |
| 幽灵依赖 | 存在（可以引用未声明的包） | 严格隔离，不允许幽灵依赖 |
| 兼容性 | 最好 | 极少数包有兼容问题 |

**推荐：** 新项目使用 pnpm，它更快、更安全、更省空间。

```bash
# 安装 pnpm
npm install -g pnpm

# 验证
pnpm --version
```

### 4.3 TypeScript 编译器（tsc）

```bash
# 全局安装（方便命令行使用）
pnpm add -g typescript

# 项目级安装（推荐，确保团队版本一致）
pnpm add -D typescript

# 验证
npx tsc --version
```
**为什么项目级安装更好？** 全局安装的 tsc 版本可能和项目需要的版本不一致。
项目级安装通过 `package.json` 锁定版本，确保所有开发者使用相同的编译器版本。

### 4.4 ts-node：直接运行 TypeScript

`tsc` 需要先编译再运行，开发时不够方便。`ts-node` 可以直接执行 `.ts` 文件：

```bash
# 安装
pnpm add -D ts-node @types/node

# 直接运行 ts 文件
npx ts-node src/index.ts
```

**更快的替代方案：tsx**

`tsx` 基于 esbuild，比 ts-node 快很多，且零配置：

```bash
pnpm add -D tsx

# 直接运行，无需任何配置
npx tsx src/index.ts

# 监听模式，文件变化自动重新运行
npx tsx watch src/index.ts
```

**为什么推荐 tsx？** ts-node 需要配置 tsconfig.json 中的模块系统才能正常工作，
ESM/CJS 的兼容问题经常让新手困惑。tsx 内部使用 esbuild 转译，几乎不需要配置。

### 4.5 VS Code 配置

VS Code 内置了 TypeScript 语言服务，开箱即用。推荐额外安装的扩展：

```json
// .vscode/extensions.json - 推荐团队统一安装的扩展
{
  "recommendations": [
    "dbaeumer.vscode-eslint",
    "esbenp.prettier-vscode",
    "ms-vscode.vscode-typescript-next"
  ]
}
```

```json
// .vscode/settings.json - 项目级 VS Code 配置
{
  "typescript.preferences.importModuleSpecifier": "relative",
  "typescript.suggest.autoImports": true,
  "editor.formatOnSave": true,
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.codeActionsOnSave": {
    "source.organizeImports": "explicit"
  }

```

**关键设置说明：**
- `importModuleSpecifier: "relative"` - 自动导入使用相对路径，避免路径别名问题
- `codeActionsOnSave` - 保存时自动整理 import 语句，删除未使用的导入
## 5. 第一个 TypeScript 程序

### 项目初始化

```bash
# 创建项目目录
mkdir my-first-ts && cd my-first-ts

# 初始化 package.json
pnpm init

# 安装 TypeScript 和相关工具
pnpm add -D typescript tsx @types/node

# 初始化 tsconfig.json
npx tsc --init
```

### 编写代码

```typescript
// src/index.ts
interface Todo {
  id: number;
  title: string;
  completed: boolean;
  createdAt: Date;
}

// 创建一个简单的 Todo 管理器
class TodoManager {
  private todos: Todo[] = [];
  private nextId = 1;

  add(title: string): Todo {
    const todo: Todo = {
      id: this.nextId++,
      title,
      completed: false,
      createdAt: new Date(),
    };
    this.todos.push(todo);
    return todo;
  }

  complete(id: number): boolean {
    const todo = this.todos.find(t => t.id === id);
    if (!todo) return false;
    todo.completed = true;
    return true;
  }

  list(filter?: 'all' | 'active' | 'completed'): Todo[] {
    switch (filter) {
      case 'active':
        return this.todos.filter(t => !t.completed);
      case 'completed':
        return this.todos.filter(t => t.completed);
      default:
        return [...this.todos];
    }
  }

  summary(): string {
    const total = this.todos.length;
    const completed = this.todos.filter(t => t.completed).length;
    return `总计: ${total}, 已完成: ${completed}, 待办: ${total - completed}`;
  }
}
// 使用
const manager = new TodoManager();
manager.add("学习 TypeScript 基础语法");
manager.add("配置开发环境");
manager.add("完成第一个项目");

manager.complete(1);

console.log("所有任务:", manager.list());
console.log("待办任务:", manager.list('active'));
console.log(manager.summary());
```

### 运行程序

```bash
# 方式 1：使用 tsx 直接运行（推荐开发时使用）
npx tsx src/index.ts

# 方式 2：先编译再运行
npx tsc
node dist/index.js
```

**这个例子展示了 TypeScript 的核心优势：**
- `Todo` 接口清晰定义了数据结构
- 方法参数和返回值都有类型标注
- `filter` 参数使用联合类型限制了合法值
- IDE 可以对 `manager` 的所有方法提供精确补全
## 6. 编译过程详解

### TypeScript 编译流程

```
.ts 源文件 → TypeScript 编译器 (tsc) → .js 输出文件
                    ↓
            类型检查（报错或通过）
                    ↓
            类型擦除（移除所有类型标注）
                    ↓
            语法降级（根据 target 转换新语法）
```

**关键理解：TypeScript 的类型只存在于编译时，运行时完全消失。**

```typescript
// 编译前 (.ts)
interface Point {
  x: number;
  y: number;
}

function distance(a: Point, b: Point): number {
  return Math.sqrt((a.x - b.x) ** 2 + (a.y - b.y) ** 2);
}

const p1: Point = { x: 0, y: 0 };
const p2: Point = { x: 3, y: 4 };
console.log(distance(p1, p2)); // 5
```

```javascript
// 编译后 (.js) - 所有类型信息被擦除
"use strict";
function distance(a, b) {
  return Math.sqrt(Math.pow(a.x - b.x, 2) + Math.pow(a.y - b.y, 2));
}
const p1 = { x: 0, y: 0 };
const p2 = { x: 3, y: 4 };
console.log(distance(p1, p2));
```

### tsconfig.json 核心配置

```json
{
  "compilerOptions": {
    // 输出目标：编译后的 JS 版本
    "target": "ES2020",

    // 模块系统
    "module": "ESNext",
    "moduleResolution": "bundler",

    // 严格模式（强烈推荐开启）
    "strict": true,

    // 输出目录
    "outDir": "./dist",
    "rootDir": "./src",

    // 生成声明文件（开发库时需要）
    "declaration": true,

    // 生成 source map（调试用）
    "sourceMap": true,

    // 允许导入 JSON
    "resolveJsonModule": true,

    // ESM 互操作
    "esModuleInterop": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
```

**为什么 strict: true 很重要？**

`strict` 是一组严格检查的总开关，包含：
- `strictNullChecks`：null/undefined 不能随意赋值
- `noImplicitAny`：不允许隐式 any
- `strictFunctionTypes`：函数参数类型严格检查
- `strictPropertyInitialization`：类属性必须初始化

新项目务必开启 `strict: true`，这能在编译时捕获大量潜在 bug。

## 7. 实际工程中的使用场景

### 前端应用开发

```typescript
// React + TypeScript：组件 Props 类型安全
interface ButtonProps {
  label: string;
  variant: 'primary' | 'secondary' | 'danger';
  disabled?: boolean;
  onClick: (event: React.MouseEvent) => void;
}

function Button({ label, variant, disabled, onClick }: ButtonProps) {
  return (
    <button className={`btn-${variant}`} disabled={disabled} onClick={onClick}>
      {label}
    </button>
  );
}
```

### 后端 API 开发

```typescript
// Express + TypeScript：类型安全的路由处理
interface CreateUserRequest {
  name: string;
  email: string;
  role: 'admin' | 'user';
}

interface ApiResponse<T> {
  success: boolean;
  data?: T;
  error?: string;
}

app.post('/users', async (req: Request<{}, {}, CreateUserRequest>, res: Response<ApiResponse<User>>) => {
  const { name, email, role } = req.body;
  const user = await userService.create({ name, email, role });
  res.json({ success: true, data: user });
});
```

### 全栈类型共享

```typescript
// shared/types.ts - 前后端共享类型定义
export interface Product {
  id: string;
  name: string;
  price: number;
  category: Category;
}

export type Category = 'electronics' | 'clothing' | 'food';

// 前端和后端都导入这个类型，确保数据结构一致
// 修改类型定义时，两端都会得到编译错误提示
```

### 工具库开发

```typescript
// 开发一个类型安全的事件总线
type EventMap = {
  'user:login': { userId: string; timestamp: number };
  'user:logout': { userId: string };
  'order:created': { orderId: string; amount: number };
};

class TypedEventBus<T extends Record<string, any>> {
  private listeners = new Map<keyof T, Set<Function>>();

  on<K extends keyof T>(event: K, handler: (payload: T[K]) => void): void {
    if (!this.listeners.has(event)) {
      this.listeners.set(event, new Set());
    }
    this.listeners.get(event)!.add(handler);
  }

  emit<K extends keyof T>(event: K, payload: T[K]): void {
    this.listeners.get(event)?.forEach(handler => handler(payload));
  }
}

const bus = new TypedEventBus<EventMap>();
bus.on('user:login', (payload) => {
  // payload 自动推断为 { userId: string; timestamp: number }
  console.log(`User ${payload.userId} logged in`);
});

// 类型错误：'order:created' 事件需要 orderId 和 amount
bus.emit('order:created', { userId: '123' }); // Error!
```
