# TypeScript 基本类型系统

TypeScript 的类型系统是它最核心的特性。理解基本类型是掌握 TypeScript 的第一步，也是后续学习高级类型的基础。

---

## 1. 原始类型 (Primitive Types)

### 1.1 string, number, boolean

这三个是最基础的类型，对应 JavaScript 的三个原始值类型。

```typescript
// 类型注解的基本语法
let username: string = "Alice";
let age: number = 25;
let isActive: boolean = true;

// TypeScript 可以自动推断类型（类型推断）
let city = "Beijing";      // 推断为 string
let count = 42;            // 推断为 number
let isDone = false;        // 推断为 boolean
```

**为什么 TypeScript 要区分这些类型？**

JavaScript 是动态类型语言，变量可以随时改变类型：

```typescript
// JavaScript 中这样写不会报错，但可能导致 bug
let value = "hello";
value = 123;  // TypeScript 报错：Type 'number' is not assignable to type 'string'
```

区分类型的好处：
- **编译时捕获错误**：避免把 number 当 string 使用
- **更好的 IDE 支持**：自动补全、重构更准确
- **代码即文档**：类型注解让代码意图更清晰

**注意：不要使用包装类型**

```typescript
// 错误示范 - 不要使用大写的包装类型
let bad: String = "hello";   // 不推荐
let bad2: Number = 42;       // 不推荐
let bad3: Boolean = true;    // 不推荐

// 正确写法 - 始终使用小写的原始类型
let good: string = "hello";  // 推荐
let good2: number = 42;      // 推荐
let good3: boolean = true;   // 推荐
```

`String` 是 JavaScript 的包装对象类型，`string` 是原始类型。在 TypeScript 中几乎永远不需要用大写形式。

### 1.2 null 和 undefined

在 JavaScript 中，`null` 和 `undefined` 都表示"没有值"，但含义不同：

```typescript
// undefined：变量已声明但未赋值，或函数没有返回值
let notAssigned: undefined = undefined;

// null：明确表示"空值"或"没有对象"
let empty: null = null;
```

**它们的区别：**
- `undefined`：系统级别的"未定义"，通常是无意的缺失
- `null`：程序级别的"空值"，通常是有意的赋值

**strictNullChecks 的意义：**

这是 `tsconfig.json` 中最重要的配置之一。

```typescript
// 当 strictNullChecks: false（不推荐）
let name: string = null;      // 不报错，但危险！
let age: number = undefined;  // 不报错，但危险！

// 当 strictNullChecks: true（强烈推荐）
let name: string = null;      // 报错！
let age: number = undefined;  // 报错！

// 如果确实需要允许 null，使用联合类型显式声明
let name: string | null = null;           // 明确表示可能为 null
let age: number | undefined = undefined;  // 明确表示可能为 undefined
```

**为什么要开启 strictNullChecks？**

```typescript
// 没有 strictNullChecks 时，这段代码编译通过但运行时崩溃
function getLength(str: string) {
  return str.length;  // 如果 str 是 null，运行时报错！
}
getLength(null);  // strictNullChecks 开启时，编译器会阻止这个调用

// 开启后，你必须处理 null 的情况
function getLength(str: string | null): number {
  if (str === null) {
    return 0;
  }
  return str.length;  // 这里 TypeScript 知道 str 一定是 string
}
```

**实际工程建议**：新项目务必开启 `strict: true`（包含 strictNullChecks）。

### 1.3 symbol 和 bigint

**Symbol - 唯一标识符：**

```typescript
// symbol 创建的值是全局唯一的
const id1: symbol = Symbol("id");
const id2: symbol = Symbol("id");
console.log(id1 === id2);  // false，即使描述相同

// unique symbol - 更精确的类型
const uniqueId: unique symbol = Symbol("unique");
// unique symbol 只能用 const 声明，因为它代表一个特定的 symbol 值
```

**Symbol 的使用场景：**
- 作为对象的属性键，避免命名冲突
- 实现迭代器协议（Symbol.iterator）
- 定义类的私有属性（在 # 语法出现之前）

```typescript
// 用 symbol 作为对象键，避免属性名冲突
const METADATA = Symbol("metadata");

interface Document {
  title: string;
  [METADATA]: { createdAt: Date };
}
```

**BigInt - 大整数：**

```typescript
// number 类型有精度限制（最大安全整数 2^53 - 1）
const maxSafe: number = Number.MAX_SAFE_INTEGER;  // 9007199254740991
const overflow: number = maxSafe + 1;  // 精度丢失！

// bigint 可以表示任意大的整数
const big: bigint = 9007199254740991n;
const bigger: bigint = BigInt("99999999999999999999");

// 注意：bigint 和 number 不能混合运算
const a: bigint = 10n;
const b: number = 20;
// const c = a + b;  // 报错！不能混合 bigint 和 number
const c = a + BigInt(b);  // 正确
```

**BigInt 使用场景：**
- 金融计算（避免浮点精度问题）
- 数据库中的大 ID
- 密码学相关计算

### 1.4 void

`void` 表示函数没有返回值（或返回 undefined）。

```typescript
// 函数没有 return 语句，返回类型为 void
function logMessage(msg: string): void {
  console.log(msg);
  // 没有 return，或者 return; 或者 return undefined;
}

// void 和 undefined 的微妙区别
// void 作为返回类型时，允许函数返回 undefined
function doNothing(): void {
  return undefined;  // 合法
}

// 但 void 类型的变量几乎没有用处
let unusable: void = undefined;  // 只能赋值 undefined
```

**void 在回调函数中的特殊行为：**

```typescript
// 当回调函数的返回类型声明为 void 时，实际返回值会被忽略
type Callback = (item: string) => void;

// 这是合法的！forEach 的回调声明返回 void，但 push 返回 number
const items: string[] = [];
const callback: Callback = (item) => items.push(item);
// push 返回数组长度（number），但因为类型声明为 void，返回值被忽略

// 这就是为什么 Array.forEach 的回调可以返回任何值
[1, 2, 3].forEach((n) => n * 2);  // 返回值被忽略，不报错
```

## 2. 数组类型

### 2.1 Array<T> vs T[] 两种写法

```typescript
// 两种写法完全等价
let numbers1: number[] = [1, 2, 3];
let numbers2: Array<number> = [1, 2, 3];

let strings1: string[] = ["a", "b", "c"];
let strings2: Array<string> = ["a", "b", "c"];
```

**如何选择？**

```typescript
// T[] 更简洁，大多数情况下推荐使用
let ids: number[] = [1, 2, 3];

// Array<T> 在复杂类型时可读性更好
let pairs: Array<[string, number]> = [["age", 25], ["score", 100]];
// 对比：[string, number][] 看起来有点混乱

// 联合类型数组时，Array<T> 更清晰
let mixed: Array<string | number> = [1, "two", 3];
// 对比：(string | number)[] 需要额外的括号
```

**常见错误：**

```typescript
// 错误：空数组没有类型信息
let items = [];  // 推断为 any[]（在 noImplicitAny 下会报错）

// 正确：声明类型
let items: string[] = [];

// 错误：数组类型不匹配
let nums: number[] = [1, 2, "3"];  // 报错："3" 不是 number
```

### 2.2 只读数组 ReadonlyArray

```typescript
// 只读数组不能被修改
const colors: ReadonlyArray<string> = ["red", "green", "blue"];
// 或者使用简写语法
const colors2: readonly string[] = ["red", "green", "blue"];

// 以下操作都会报错
colors.push("yellow");     // 报错：ReadonlyArray 没有 push 方法
colors[0] = "purple";      // 报错：只读属性不能赋值
colors.splice(0, 1);       // 报错：ReadonlyArray 没有 splice 方法

// 但可以读取
console.log(colors[0]);    // "red" - 合法
console.log(colors.length); // 3 - 合法
const mapped = colors.map(c => c.toUpperCase());  // 合法，返回新数组
```

**使用场景：**

```typescript
// 函数参数使用只读数组，保证函数不会修改传入的数组
function getFirst(items: readonly string[]): string | undefined {
  // items.push("new");  // 报错！不能修改
  return items[0];       // 只能读取
}

// 配置项通常应该是只读的
const CONFIG_KEYS: readonly string[] = ["host", "port", "database"];
```

### 2.3 元组 Tuple

元组是固定长度、每个位置有特定类型的数组。

```typescript
// 元组：明确每个位置的类型
let point: [number, number] = [10, 20];
let entry: [string, number] = ["age", 25];

// 访问元素时，TypeScript 知道每个位置的类型
const name = entry[0];   // 类型是 string
const value = entry[1];  // 类型是 number

// 解构赋值
const [x, y] = point;    // x: number, y: number
```

**什么时候用元组而不是数组？**

```typescript
// 当数据有固定结构且位置有含义时用元组
type HttpResponse = [number, string, object]; // [状态码, 消息, 数据]
const response: HttpResponse = [200, "OK", { id: 1 }];

// React 的 useState 返回的就是元组
// const [state, setState] = useState(initialValue);
// 类型是 [S, Dispatch<SetStateAction<S>>]

// 带标签的元组（提高可读性）
type Range = [start: number, end: number];
const range: Range = [0, 100];
```

## 3. 对象类型

### 3.1 对象字面量类型

```typescript
// 直接定义对象的形状
let user: { name: string; age: number } = {
  name: "Alice",
  age: 30
};

// 嵌套对象
let config: {
  host: string;
  port: number;
  database: {
    name: string;
    connection: string;
  };
} = {
  host: "localhost",
  port: 3000,
  database: {
    name: "mydb",
    connection: "postgresql://..."
  }
};
```

### 3.2 可选属性

```typescript
// ? 表示属性可以不存在
interface UserProfile {
  name: string;
  email: string;
  phone?: string;      // 可选：类型是 string | undefined
  avatar?: string;     // 可选
}

// 合法：phone 和 avatar 可以省略
const profile: UserProfile = {
  name: "Bob",
  email: "bob@example.com"
};

// 使用可选属性时需要检查
function formatPhone(user: UserProfile): string {
  if (user.phone) {
    return `Tel: ${user.phone}`;
  }
  return "No phone";
}
```

### 3.3 只读属性

```typescript
interface Point {
  readonly x: number;
  readonly y: number;
}

const origin: Point = { x: 0, y: 0 };
origin.x = 1; // Error: Cannot assign to 'x' because it is a read-only property

// 实际工程场景：配置对象应该是只读的
interface AppConfig {
  readonly apiUrl: string;
  readonly timeout: number;
  readonly features: readonly string[];
}

// 为什么用 readonly？
// 1. 防止意外修改共享状态
// 2. 让代码意图更清晰：这个值初始化后不应该变
// 3. 在 React 中，Props 应该是只读的
```

## 4. 特殊类型

### 4.1 any - 逃生舱

```typescript
// any 关闭了类型检查，任何操作都不会报错
let value: any = "hello";
value = 42;
value = { foo: "bar" };
value.nonExistentMethod(); // 不报错！运行时才会崩溃

// 什么时候不得不用 any？
// 1. 迁移 JavaScript 项目时的过渡
// 2. 第三方库没有类型定义
// 3. 确实无法确定类型的动态数据

// 为什么要避免 any？
// 它让 TypeScript 的类型检查完全失效
let x: any = "hello";
let y: number = x; // 不报错，但 y 实际上是 string！
```

### 4.2 unknown - 安全的 any

```typescript
// unknown 是类型安全的 any：你可以赋任何值，但使用前必须检查类型
let value: unknown = "hello";
value = 42;
value = { foo: "bar" };

// 不能直接使用 unknown 类型的值
// value.foo;          // Error!
// value.toString();   // Error!
// value + 1;          // Error!

// 必须先收窄类型
if (typeof value === "string") {
  console.log(value.toUpperCase()); // OK，这里 value 是 string
}

if (typeof value === "number") {
  console.log(value + 1); // OK，这里 value 是 number
}

// 实际工程场景：处理外部输入
async function fetchData(url: string): Promise<unknown> {
  const response = await fetch(url);
  return response.json(); // 返回 unknown，调用方必须验证
}

// 调用方必须做类型检查
const data = await fetchData("/api/users");
if (Array.isArray(data) && data.every(item => typeof item.name === "string")) {
  // 现在可以安全使用
}
```

**为什么设计 unknown？**

TypeScript 3.0 引入 unknown 是为了提供一个比 any 更安全的"我不知道类型"选项。
any 是"我不在乎类型检查"，unknown 是"我还不知道类型，但我承诺使用前会检查"。

### 4.3 never - 不可能的类型

```typescript
// never 表示永远不会出现的值

// 场景 1：抛出异常的函数（永远不会正常返回）
function throwError(message: string): never {
  throw new Error(message);
}

// 场景 2：无限循环
function infiniteLoop(): never {
  while (true) {
    // 永远不会结束
  }
}

// 场景 3：穷尽检查（最重要的用法！）
type Shape = "circle" | "square" | "triangle";

function getArea(shape: Shape): number {
  switch (shape) {
    case "circle":
      return Math.PI * 10 * 10;
    case "square":
      return 10 * 10;
    case "triangle":
      return (10 * 10) / 2;
    default:
      // 如果所有 case 都覆盖了，shape 在这里是 never
      const _exhaustive: never = shape;
      return _exhaustive;
  }
}

// 如果以后有人添加了新的 Shape 类型但忘了处理：
// type Shape = "circle" | "square" | "triangle" | "pentagon";
// default 分支会报错：Type 'string' is not assignable to type 'never'
// 这就是编译时的穷尽检查！
```

### 4.4 object vs Object vs {}

```typescript
// object（小写）：非原始类型，即不是 string/number/boolean/null/undefined/symbol/bigint
let obj: object = { name: "Alice" }; // OK
obj = [1, 2, 3];                     // OK（数组也是对象）
obj = () => {};                       // OK（函数也是对象）
obj = "hello";                        // Error: 原始类型不行

// Object（大写）：几乎所有值（有 toString/valueOf 等方法的值）
// 不推荐使用，因为它太宽泛了
let val: Object = "hello"; // OK（string 有 toString）
val = 42;                  // OK
val = null;                // Error（strictNullChecks 下）

// {}（空对象类型）：和 Object 类似，表示任何非 null/undefined 的值
let empty: {} = "hello"; // OK
empty = 42;              // OK
empty = null;            // Error

// 实际建议：
// - 需要"任何对象"时用 object 或 Record<string, unknown>
// - 需要"任何值"时用 unknown
// - 避免使用 Object 和 {}
```

## 5. 类型断言

### 5.1 as 语法

```typescript
// 类型断言告诉编译器"我比你更了解这个值的类型"
const input = document.getElementById("username") as HTMLInputElement;
input.value = "hello"; // 没有断言的话，类型是 HTMLElement | null

// 什么时候该用？
// 1. DOM 操作（你知道元素的具体类型）
// 2. 第三方库返回过于宽泛的类型
// 3. 类型收窄无法覆盖的场景

// 什么时候不该用？
// 当你不确定类型时！断言不做运行时检查
const data = JSON.parse(response) as User; // 危险！如果数据格式不对呢？

// 更安全的做法：运行时验证
function isUser(data: unknown): data is User {
  return (
    typeof data === "object" &&
    data !== null &&
    "name" in data &&
    "email" in data
  );
}
```

### 5.2 非空断言 !

```typescript
// ! 告诉编译器"这个值一定不是 null/undefined"
function getLength(str: string | null): number {
  // 你确定 str 不为 null 时可以用
  return str!.length;
}

// 实际场景：DOM 元素你确定存在
const app = document.getElementById("app")!;

// 警告：非空断言是危险的，如果值确实是 null 会运行时崩溃
// 更好的做法是用条件检查
const app2 = document.getElementById("app");
if (!app2) throw new Error("App element not found");
// 这之后 app2 自动收窄为 HTMLElement（非 null）
```

## 6. 字面量类型

```typescript
// 字面量类型：值本身就是类型
let direction: "north" | "south" | "east" | "west";
direction = "north"; // OK
direction = "up";    // Error: Type '"up"' is not assignable

// 数字字面量
type DiceRoll = 1 | 2 | 3 | 4 | 5 | 6;
let roll: DiceRoll = 3; // OK
roll = 7;               // Error

// 为什么字面量类型有用？
// 它让你精确描述值的范围，比 string/number 更严格

// 实际工程场景：HTTP 方法
type HttpMethod = "GET" | "POST" | "PUT" | "DELETE" | "PATCH";

function request(url: string, method: HttpMethod) {
  // method 只能是这 5 个值之一
}

request("/api/users", "GET");    // OK
request("/api/users", "REMOVE"); // Error!
```

### const 断言

```typescript
// 普通声明：TypeScript 推断为宽泛类型
let color = "red";        // 类型是 string
const colors = ["red", "green", "blue"]; // 类型是 string[]

// as const：推断为最窄的字面量类型
const color2 = "red" as const;           // 类型是 "red"
const colors2 = ["red", "green", "blue"] as const;
// 类型是 readonly ["red", "green", "blue"]

// 实际工程场景：配置对象
const CONFIG = {
  API_URL: "https://api.example.com",
  TIMEOUT: 5000,
  RETRIES: 3,
} as const;
// CONFIG.API_URL 的类型是 "https://api.example.com"（不是 string）
// CONFIG.TIMEOUT 的类型是 5000（不是 number）
// 所有属性自动变为 readonly

// 从 as const 对象提取类型
type ConfigKeys = keyof typeof CONFIG; // "API_URL" | "TIMEOUT" | "RETRIES"
type ApiUrl = typeof CONFIG.API_URL;   // "https://api.example.com"
```

## 7. 枚举 Enum

### 7.1 数字枚举

```typescript
// 默认从 0 开始递增
enum Direction {
  Up,    // 0
  Down,  // 1
  Left,  // 2
  Right  // 3
}

let dir: Direction = Direction.Up;
console.log(dir); // 0

// 可以指定起始值
enum HttpStatus {
  OK = 200,
  Created = 201,
  BadRequest = 400,
  NotFound = 404,
  InternalError = 500
}
```

### 7.2 字符串枚举

```typescript
enum LogLevel {
  Debug = "DEBUG",
  Info = "INFO",
  Warn = "WARN",
  Error = "ERROR"
}

function log(level: LogLevel, message: string) {
  console.log(`[${level}] ${message}`);
}

log(LogLevel.Info, "Server started"); // [INFO] Server started
```

### 7.3 const enum

```typescript
// const enum 在编译时被完全内联，不生成运行时代码
const enum Color {
  Red = "RED",
  Green = "GREEN",
  Blue = "BLUE"
}

const myColor = Color.Red;
// 编译后：const myColor = "RED";
// 普通 enum 编译后会生成一个对象，const enum 直接替换为值
```

### 7.4 枚举 vs 联合类型

```typescript
// 方式 1：枚举
enum Status {
  Active = "active",
  Inactive = "inactive",
  Pending = "pending"
}

// 方式 2：联合类型（越来越多人推荐这种）
type Status2 = "active" | "inactive" | "pending";

// 为什么有人建议不用 enum？
// 1. enum 生成额外的运行时代码（增加包体积）
// 2. enum 是 TypeScript 独有语法，不是标准 JS
// 3. 联合类型更简洁，且 tree-shaking 友好
// 4. 数字 enum 有反向映射的坑

// 什么时候还是用 enum？
// 1. 需要运行时遍历所有值时（Object.values(Status)）
// 2. 需要在运行时做反向查找时
// 3. 团队约定使用 enum 时

// 推荐做法：大多数场景用联合类型 + as const 对象
const STATUS = {
  Active: "active",
  Inactive: "inactive",
  Pending: "pending",
} as const;

type Status3 = typeof STATUS[keyof typeof STATUS];
// "active" | "inactive" | "pending"
```
