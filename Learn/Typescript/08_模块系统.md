# 第八章：模块系统

模块系统是 JavaScript/TypeScript 工程化的基石。理解模块系统的演进历史和当前最佳实践，对于构建可维护的大型项目至关重要。

---

## 1. 模块系统演进历史

### 1.1 早期的全局变量时代

在模块系统出现之前，JavaScript 通过全局变量共享代码：

```javascript
// math.js
var MathUtils = {
  add: function(a, b) { return a + b; },
  multiply: function(a, b) { return a * b; }
};

// app.js — 依赖 math.js 先加载
var result = MathUtils.add(1, 2);
```

问题显而易见：命名冲突、依赖顺序不可控、无法按需加载。

### 1.2 CommonJS (CJS)

Node.js 采用的模块规范，同步加载，适合服务端：

```javascript
// math.js
function add(a, b) { return a + b; }
function multiply(a, b) { return a * b; }

module.exports = { add, multiply };
// 或者
exports.add = add;
exports.multiply = multiply;

// app.js
const { add, multiply } = require('./math');
console.log(add(1, 2)); // 3
```

**设计特点：**
- `require` 是同步的，模块在首次加载时执行并缓存
- `module.exports` 是真正的导出对象，`exports` 只是它的引用
- 运行时解析，可以条件性加载

```javascript
// 条件加载 — CJS 独有能力
if (process.env.NODE_ENV === 'production') {
  module.exports = require('./optimized');
} else {
  module.exports = require('./debug');
}
```

### 1.3 AMD (Asynchronous Module Definition)

为浏览器设计的异步模块规范，代表实现是 RequireJS：

```javascript
// 定义模块
define('math', [], function() {
  return {
    add: function(a, b) { return a + b; }
  };
});

// 使用模块
require(['math'], function(math) {
  console.log(math.add(1, 2));
});
```

AMD 解决了浏览器环境的异步加载问题，但语法冗长，开发体验差。现在已基本被淘汰。

### 1.4 UMD (Universal Module Definition)

兼容 CJS 和 AMD 的通用格式：

```javascript
(function(root, factory) {
  if (typeof define === 'function' && define.amd) {
    define(['dependency'], factory);
  } else if (typeof module === 'object' && module.exports) {
    module.exports = factory(require('dependency'));
  } else {
    root.MyModule = factory(root.Dependency);
  }
})(this, function(dependency) {
  return { /* 模块内容 */ };
});
```

UMD 常见于需要同时支持多种环境的库（如 lodash、moment）。现在主要用于需要通过 `<script>` 标签直接引入的场景。

### 1.5 ES Modules (ESM)

JavaScript 语言层面的官方模块标准（ES2015+）：

```javascript
// math.ts
export function add(a: number, b: number): number {
  return a + b;
}

export function multiply(a: number, b: number): number {
  return a * b;
}

// app.ts
import { add, multiply } from './math';
console.log(add(1, 2));
```

**ESM 的核心优势：**

| 特性 | CJS | ESM |
|------|-----|-----|
| 加载方式 | 同步 | 异步（可静态分析） |
| 绑定方式 | 值拷贝 | 活绑定（live binding） |
| 静态分析 | 不可能 | 可以（Tree Shaking） |
| 顶层 await | 不支持 | 支持 |
| 浏览器原生 | 不支持 | 支持 |

### 1.6 为什么现在推荐 ESM

1. **标准化**：ESM 是 ECMAScript 规范的一部分，不是社区方案
2. **静态结构**：导入导出在编译时确定，支持 Tree Shaking
3. **活绑定**：导出的是引用而非值拷贝，模块间状态同步
4. **浏览器原生支持**：`<script type="module">` 无需打包即可运行
5. **Node.js 全面支持**：Node 12+ 支持 ESM，Node 22 已默认 ESM

```javascript
// ESM 的活绑定特性
// counter.js
export let count = 0;
export function increment() { count++; }

// app.js
import { count, increment } from './counter';
console.log(count); // 0
increment();
console.log(count); // 1 — 活绑定，值同步更新

// 如果是 CJS，count 始终是 0（值拷贝）
```

## 2. TypeScript 中的模块

### 2.1 import/export 语法

TypeScript 完全支持 ESM 语法，并在此基础上增加了类型相关的能力：

```typescript
// === 命名导出 ===
export interface User {
  id: number;
  name: string;
  email: string;
}

export type UserRole = 'admin' | 'editor' | 'viewer';

export function createUser(name: string, email: string): User {
  return { id: Date.now(), name, email };
}

export const MAX_USERS = 1000;

// === 命名导入 ===
import { User, UserRole, createUser, MAX_USERS } from './user';

// === 重命名导入 ===
import { createUser as makeUser } from './user';

// === 整体导入（命名空间导入） ===
import * as UserModule from './user';
const user = UserModule.createUser('Alice', 'alice@example.com');
```

### 类型导入 import type

```typescript
// 仅导入类型（编译后完全消失，不产生运行时代码）
import type { User, UserRole } from './user';

// 内联类型导入
import { createUser, type User } from './user';

// 为什么用 import type？
// 1. 明确表达意图：这个导入只用于类型检查
// 2. 避免循环依赖问题（类型导入不产生运行时依赖）
// 3. 帮助打包工具做 tree-shaking
```

### 动态导入

```typescript
// 动态导入返回 Promise，支持代码分割
async function loadChart() {
  const { Chart } = await import('./chart');
  return new Chart();
}

// 条件导入
async function getLogger(env: string) {
  if (env === 'production') {
    return await import('./prod-logger');
  }
  return await import('./dev-logger');
}
```

## 3. 模块解析策略

```json
// tsconfig.json
{
  "compilerOptions": {
    // 现代项目推荐 "bundler"
    "moduleResolution": "bundler",

    // 路径映射
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"],
      "@components/*": ["src/components/*"],
      "@utils/*": ["src/utils/*"]
    }
  }
}
```

**为什么模块解析经常出问题？**

TypeScript 的模块解析需要和运行时/打包工具的解析保持一致。常见问题：
- `paths` 只影响 TypeScript 编译，运行时还需要 webpack alias 或 tsconfig-paths
- `.js` 扩展名问题：ESM 要求写扩展名，但源码是 `.ts`
- 解决方案：使用 `moduleResolution: "bundler"` 配合现代打包工具

## 4. 命名空间 vs 模块

```typescript
// 命名空间（不推荐在现代项目中使用）
namespace Validation {
  export interface Validator {
    validate(value: string): boolean;
  }

  export class EmailValidator implements Validator {
    validate(value: string) {
      return value.includes('@');
    }
  }
}

// 为什么现代项目不推荐命名空间？
// 1. 文件本身就是模块，不需要额外的命名空间包裹
// 2. 命名空间不支持 tree-shaking
// 3. 与 ES Modules 混用会造成困惑
// 4. 唯一合理的使用场景：全局类型声明（如 declare global）
```

## 5. 模块增强与全局声明

```typescript
// 扩展已有模块的类型
declare module 'express' {
  interface Request {
    user?: { id: string; role: string };
  }
}

// 声明全局变量
declare global {
  interface Window {
    __APP_CONFIG__: { apiUrl: string; version: string };
  }
}

// 声明非 JS 模块
declare module '*.css' {
  const classes: Record<string, string>;
  export default classes;
}

declare module '*.svg' {
  const content: string;
  export default content;
}
```

## 6. 实际工程案例

### Barrel 文件（index.ts 重新导出）

```typescript
// src/components/index.ts — barrel 文件
export { Button } from './Button';
export { Input } from './Input';
export { Modal } from './Modal';
export type { ButtonProps } from './Button';

// 使用方可以从一个入口导入
import { Button, Input, Modal } from '@/components';

// 注意：barrel 文件可能影响 tree-shaking
// 如果打包工具不够智能，导入一个组件可能打包所有组件
// 解决方案：确保使用支持 sideEffects: false 的打包工具
```

### 循环依赖的解决

```typescript
// 问题：A 导入 B，B 导入 A
// 解决方案 1：提取共享类型到第三个文件
// types.ts
export interface SharedType { /* ... */ }

// 解决方案 2：使用 import type（类型导入不产生运行时循环）
import type { B } from './b'; // 不会造成运行时循环
```
