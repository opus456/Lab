# 第二章：接口与类型别名

## 1. 接口 (Interface)

### 1.1 基本语法和使用

接口是 TypeScript 中定义对象形状（shape）的核心方式。它描述了一个对象应该具有哪些属性以及这些属性的类型。

```typescript
// 定义一个用户接口
interface User {
  name: string;
  age: number;
  email: string;
}

// 使用接口约束对象
const user: User = {
  name: "张三",
  age: 28,
  email: "zhangsan@example.com"
};

// 错误示例：缺少必要属性会报错
// const badUser: User = { name: "李四" }; // Error: 缺少 age 和 email
```

**设计原因**：接口的核心价值在于"契约"——它定义了代码之间的约定。当团队协作时，接口让每个人都清楚数据的结构，编译器会自动检查是否遵守了约定。

### 1.2 可选属性和只读属性

```typescript
interface Config {
  // 只读属性：初始化后不可修改
  readonly id: string;
  readonly createdAt: Date;

  // 必选属性
  host: string;
  port: number;

  // 可选属性：可以存在也可以不存在
  database?: string;
  password?: string;
  ssl?: boolean;
}

const config: Config = {
  id: "cfg-001",
  createdAt: new Date(),
  host: "localhost",
  port: 5432,
  // database, password, ssl 可以省略
};

// config.id = "cfg-002"; // Error: 无法分配到 "id" ，因为它是只读属性
config.host = "192.168.1.100"; // OK：非只读属性可以修改
```

**工程场景**：`readonly` 常用于配置对象、ID字段等不应被意外修改的数据。可选属性适合有默认值或非必填的配置项。

```typescript
// 实际工程中的常见模式：带默认值的配置
function createConnection(options: Config) {
  const finalConfig = {
    database: "postgres",  // 默认值
    ssl: false,            // 默认值
    ...options,            // 用户传入的值覆盖默认值
  };
  // ...
}
```

### 1.3 函数类型接口

接口不仅可以描述对象，还可以描述函数的形状：

```typescript
// 方式一：使用接口描述函数类型
interface SearchFunc {
  (source: string, keyword: string): boolean;
}

const search: SearchFunc = (source, keyword) => {
  return source.includes(keyword);
};

// 方式二：带有属性的函数（可调用对象）
interface Counter {
  (): number;          // 调用签名
  count: number;       // 属性
  reset(): void;       // 方法
}

function createCounter(): Counter {
  const fn = function () {
    fn.count++;
    return fn.count;
  } as Counter;
  fn.count = 0;
  fn.reset = () => { fn.count = 0; };
  return fn;
}

const counter = createCounter();
counter();       // 1
counter();       // 2
counter.count;   // 2
counter.reset();
counter();       // 1
```

**设计原因**：JavaScript 中函数是一等公民，函数本身也可以有属性（如 jQuery 的 `$` 既是函数又有 `.ajax` 等属性）。TypeScript 的函数类型接口完美描述了这种模式。

### 1.4 可索引类型（索引签名）

索引签名允许你描述可以通过索引访问的类型，比如数组或字典：

```typescript
// 字符串索引签名：字典/映射类型
interface StringDictionary {
  [key: string]: string;
}

const headers: StringDictionary = {
  "Content-Type": "application/json",
  "Authorization": "Bearer token123",
};

// 数字索引签名：类数组类型
interface NumberArray {
  [index: number]: string;
}

const fruits: NumberArray = ["apple", "banana", "cherry"];

// 混合：固定属性 + 索引签名
interface CSSStyles {
  display?: string;
  position?: string;
  [property: string]: string | undefined; // 索引签名必须兼容已知属性的类型
}

const styles: CSSStyles = {
  display: "flex",
  position: "relative",
  "margin-top": "10px",
  "padding-left": "20px",
};
```

**重要规则**：当同时存在固定属性和索引签名时，固定属性的类型必须是索引签名返回类型的子类型。

```typescript
// 错误示例
// interface Bad {
//   name: number;           // Error!
//   [key: string]: string;  // 索引签名要求所有属性值为 string
// }

// 正确：使用联合类型
interface Good {
  name: string;
  age: number;
  [key: string]: string | number; // 兼容 name 和 age 的类型
}
```

### 1.5 接口继承（extends）

接口可以继承其他接口，实现类型的组合和复用：

```typescript
// 基础接口
interface Timestamps {
  createdAt: Date;
  updatedAt: Date;
}

interface SoftDelete {
  deletedAt: Date | null;
  isDeleted: boolean;
}

// 单继承
interface BaseEntity extends Timestamps {
  id: string;
}

// 多继承：一个接口可以继承多个接口
interface Article extends BaseEntity, SoftDelete {
  title: string;
  content: string;
  author: string;
  tags: string[];
}

// 使用时必须满足所有继承链上的属性
const article: Article = {
  id: "art-001",
  createdAt: new Date(),
  updatedAt: new Date(),
  deletedAt: null,
  isDeleted: false,
  title: "TypeScript 接口详解",
  content: "...",
  author: "张三",
  tags: ["typescript", "tutorial"],
};
```

**工程场景**：接口继承在数据库实体设计中非常常见。`Timestamps`、`SoftDelete` 这类通用字段抽成基础接口，业务实体继承它们，避免重复定义。

### 1.6 接口合并（声明合并）

TypeScript 中同名接口会自动合并，这是接口独有的特性：

```typescript
// 第一次声明
interface Window {
  title: string;
}

// 第二次声明（自动合并）
interface Window {
  appVersion: string;
}

// 合并后等价于：
// interface Window {
//   title: string;
//   appVersion: string;
// }

const win: Window = {
  title: "My App",
  appVersion: "1.0.0",
};
```

**为什么 TypeScript 支持这个特性？**

这不是偶然设计，而是为了解决实际问题：

1. **扩展第三方库的类型**：当你使用一个库但需要给它的类型添加属性时，声明合并让你无需修改源码：

```typescript
// 扩展 Express 的 Request 类型，添加自定义属性
declare namespace Express {
  interface Request {
    userId?: string;
    role?: "admin" | "user";
  }
}

// 现在中间件中可以安全地使用
// req.userId = "user-123";  // 不再报错
```

2. **Polyfill 和环境扩展**：为全局对象添加新 API 的类型定义。

3. **渐进式类型定义**：大型项目中可以在不同文件中逐步完善同一个接口。

## 2. 类型别名 (Type Alias)

### 2.1 基本语法

`type` 关键字用于给类型起一个新名字：

```typescript
// 基本类型别名
type ID = string;
type Age = number;
type Callback = (data: string) => void;

// 对象类型别名
type Point = {
  x: number;
  y: number;
};

// 元组类型
type Coordinate = [number, number];
type RGB = [number, number, number];

// 字面量类型
type Direction = "up" | "down" | "left" | "right";
type HttpMethod = "GET" | "POST" | "PUT" | "DELETE" | "PATCH";

// 使用
const move = (direction: Direction, distance: number) => {
  console.log(`Moving ${direction} by ${distance}`);
};

move("up", 10);    // OK
// move("forward", 5); // Error: "forward" 不在 Direction 中
```

### 2.2 与接口的区别和联系

| 特性 | interface | type |
|------|-----------|------|
| 对象类型 | 支持 | 支持 |
| 函数类型 | 支持 | 支持 |
| 联合类型 | 不支持 | 支持 |
| 交叉类型 | 用 extends | 用 & |
| 声明合并 | 支持 | 不支持 |
| 元组 | 不直接支持 | 支持 |
| 基本类型别名 | 不支持 | 支持 |
| implements | 支持 | 部分支持 |
| 计算属性 | 不支持 | 支持（映射类型） |

```typescript
// interface 和 type 都能描述对象
interface UserInterface {
  name: string;
  age: number;
}

type UserType = {
  name: string;
  age: number;
};

// 两者在大多数场景下可以互换使用
const user1: UserInterface = { name: "A", age: 20 };
const user2: UserType = { name: "B", age: 25 };
```

### 2.3 联合类型 |

联合类型表示一个值可以是几种类型之一：

```typescript
// 基础联合类型
type StringOrNumber = string | number;

function formatId(id: StringOrNumber): string {
  // 类型收窄：在使用前需要判断具体类型
  if (typeof id === "string") {
    return id.toUpperCase();
  }
  return id.toString().padStart(6, "0");
}

formatId("abc");  // "ABC"
formatId(42);     // "000042"

// 联合类型在实际工程中的应用
type Result<T> = 
  | { success: true; data: T }
  | { success: false; error: string };

function fetchUser(id: string): Result<User> {
  try {
    // ... 获取用户
    return { success: true, data: { name: "张三", age: 28, email: "z@e.com" } };
  } catch (e) {
    return { success: false, error: "用户不存在" };
  }
}
```

### 2.4 交叉类型 &

交叉类型将多个类型合并为一个类型，新类型拥有所有类型的属性：

```typescript
type HasName = { name: string };
type HasAge = { age: number };
type HasEmail = { email: string };

// 交叉类型：同时具有所有属性
type Person = HasName & HasAge & HasEmail;

const person: Person = {
  name: "张三",
  age: 28,
  email: "zhangsan@example.com",
};

// 交叉类型 vs 接口继承：效果类似
interface PersonInterface extends HasName, HasAge, HasEmail {}

// 交叉类型处理冲突的方式
type A = { value: string };
type B = { value: number };
type C = A & B;
// C 的 value 类型是 string & number = never（不可能同时是两种类型）
// 这意味着 C 类型的变量实际上无法被创建
```

### 2.5 什么时候用 interface 什么时候用 type

**工程实践建议**：

```typescript
// 1. 定义对象的公共 API（尤其是库的导出类型）→ 用 interface
//    原因：支持声明合并，用户可以扩展
interface PluginOptions {
  name: string;
  version: string;
}

// 2. 联合类型、交叉类型、元组、映射类型 → 用 type
//    原因：interface 做不到
type Status = "loading" | "success" | "error";
type Pair<T> = [T, T];

// 3. React 组件 Props → 两者都行，团队统一即可
//    社区趋势：越来越多项目统一用 type
type ButtonProps = {
  label: string;
  onClick: () => void;
  variant?: "primary" | "secondary";
};

// 4. 需要 extends 继承体系 → 用 interface
interface Shape {
  area(): number;
}
interface Circle extends Shape {
  radius: number;
}

// 5. 简单的函数类型 → 用 type（更简洁）
type Comparator<T> = (a: T, b: T) => number;
```

**总结原则**：如果不确定，用 `interface` 定义对象类型，用 `type` 定义其他一切。团队内保持一致比选哪个更重要。

## 3. 结构化类型系统详解

### 鸭子类型

TypeScript 使用结构化类型系统（Structural Type System），也叫"鸭子类型"：如果它走起来像鸭子、叫起来像鸭子，那它就是鸭子。

```typescript
interface Point {
  x: number;
  y: number;
}

function printPoint(p: Point) {
  console.log(`(${p.x}, ${p.y})`);
}

// 这个对象没有声明实现 Point，但结构匹配就行
const myObj = { x: 10, y: 20, z: 30 };
printPoint(myObj); // OK！myObj 有 x 和 y，满足 Point 的结构

// 对比名义类型系统（如 Java/C#）：
// 在 Java 中，即使结构完全一样，不 implements Point 就不行
// TypeScript 选择结构化类型是为了匹配 JavaScript 的灵活性
```

### 为什么选择结构化类型？

JavaScript 本身就是鸭子类型的：
```javascript
// JS 中你从来不关心对象"是什么类"，只关心它"有什么属性"
function serialize(obj) {
  return JSON.stringify(obj); // 任何对象都行
}
```

TypeScript 的结构化类型系统是对 JavaScript 这种使用模式的形式化。如果用名义类型，大量现有 JS 代码模式将无法表达。

### 多余属性检查

```typescript
interface Config {
  host: string;
  port: number;
}

// 直接赋值字面量时，TypeScript 会做多余属性检查
const config: Config = {
  host: "localhost",
  port: 3000,
  timeout: 5000  // Error! Object literal may only specify known properties
};

// 但通过变量赋值就不会检查（结构化类型兼容）
const obj = { host: "localhost", port: 3000, timeout: 5000 };
const config2: Config = obj; // OK！

// 为什么这样设计？
// 直接写字面量时，多余属性通常是拼写错误（如 timout 写成 timeout）
// 通过变量传递时，对象可能在其他地方有用途，多余属性是正常的
```

## 4. 联合类型与交叉类型深入

### 可辨识联合（Discriminated Unions）

这是 TypeScript 中最重要的模式之一，用于建模"多种可能状态"。

```typescript
// 用一个共同的字面量属性（判别属性）来区分不同的类型

// API 请求状态
type RequestState<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "error"; error: string };

function renderUser(state: RequestState<User>) {
  switch (state.status) {
    case "idle":
      return "Click to load";
    case "loading":
      return "Loading...";
    case "success":
      return `Hello, ${state.data.name}`; // TypeScript 知道这里有 data
    case "error":
      return `Error: ${state.error}`;     // TypeScript 知道这里有 error
  }
}

// 为什么这个模式如此重要？
// 1. 编译器强制你处理所有状态
// 2. 每个分支中自动收窄类型，访问正确的属性
// 3. 添加新状态时，未处理的地方会报错
```

```typescript
// 实际工程：Redux Action 设计
type Action =
  | { type: "ADD_TODO"; payload: { text: string } }
  | { type: "TOGGLE_TODO"; payload: { id: number } }
  | { type: "DELETE_TODO"; payload: { id: number } };

function reducer(state: TodoState, action: Action): TodoState {
  switch (action.type) {
    case "ADD_TODO":
      // action.payload 自动推断为 { text: string }
      return { ...state, todos: [...state.todos, { text: action.payload.text }] };
    case "TOGGLE_TODO":
      // action.payload 自动推断为 { id: number }
      return { ...state };
    case "DELETE_TODO":
      return { ...state };
  }
}
```

### 交叉类型的实际应用

```typescript
// 交叉类型用于组合多个类型（Mixin 模式）
type Timestamped = {
  createdAt: Date;
  updatedAt: Date;
};

type SoftDeletable = {
  deletedAt: Date | null;
  isDeleted: boolean;
};

type Identifiable = {
  id: string;
};

// 组合出完整的实体类型
type BaseEntity = Identifiable & Timestamped & SoftDeletable;

interface User extends BaseEntity {
  name: string;
  email: string;
}

// User 现在有 id, createdAt, updatedAt, deletedAt, isDeleted, name, email
```

## 5. 实际工程案例

### API 响应类型设计

```typescript
// 统一的 API 响应格式
interface ApiResponse<T> {
  code: number;
  message: string;
  data: T;
  timestamp: number;
}

// 分页响应
interface PaginatedData<T> {
  items: T[];
  total: number;
  page: number;
  pageSize: number;
  hasMore: boolean;
}

// 组合使用
type UserListResponse = ApiResponse<PaginatedData<User>>;

// 错误响应
interface ApiError {
  code: number;
  message: string;
  details?: Record<string, string[]>; // 字段级错误
}
```

### 表单数据类型

```typescript
// 表单字段状态
interface FieldState<T> {
  value: T;
  error: string | null;
  touched: boolean;
  dirty: boolean;
}

// 从数据类型自动生成表单状态类型
type FormState<T> = {
  [K in keyof T]: FieldState<T[K]>;
};

interface LoginData {
  email: string;
  password: string;
  rememberMe: boolean;
}

// LoginFormState 的每个字段都有 value, error, touched, dirty
type LoginFormState = FormState<LoginData>;
```

### 组件 Props 类型设计

```typescript
// 基础按钮 Props
interface BaseButtonProps {
  size?: "sm" | "md" | "lg";
  disabled?: boolean;
  loading?: boolean;
  className?: string;
}

// 不同变体的按钮有不同的必填属性
type ButtonProps =
  | (BaseButtonProps & { variant: "link"; href: string; onClick?: never })
  | (BaseButtonProps & { variant: "button"; onClick: () => void; href?: never })
  | (BaseButtonProps & { variant: "submit"; form: string });

// 使用时 TypeScript 会根据 variant 要求不同的属性
// <Button variant="link" href="/about" />        OK
// <Button variant="link" onClick={() => {}} />   Error: onClick 不能用于 link
// <Button variant="button" onClick={() => {}} /> OK
```
