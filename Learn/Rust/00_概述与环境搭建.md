# 第零章：Rust 概述与环境搭建

> **难度：入门** | 本章没有难点，但理解 Rust 的设计哲学对后续学习至关重要。

---

## 1. Rust 是什么

### 历史背景

Rust 由 Mozilla 工程师 Graydon Hoare 于 2006 年作为个人项目启动，2009 年 Mozilla 开始赞助，2015 年发布 1.0 稳定版。2021 年成立独立的 Rust 基金会（成员包括 AWS、Google、Microsoft、Meta 等）。

### 为什么创造 Rust

Mozilla 在开发 Firefox 浏览器引擎时面临一个根本矛盾：

- **C/C++**：性能极好，但内存安全问题层出不穷（悬垂指针、缓冲区溢出、数据竞争）
- **Java/Go**：内存安全，但有 GC 停顿，不适合浏览器引擎这种延迟敏感的场景

Rust 的目标：**在没有垃圾回收的前提下，通过编译时检查保证内存安全和线程安全。**

### 核心设计哲学

```
性能 ≈ C/C++
安全 ≈ Java/Go（甚至更好，编译时保证）
并发 = 无数据竞争（编译器强制检查）
```

Rust 的核心创新是**所有权系统（Ownership System）**：
- 每个值有且只有一个所有者
- 所有者离开作用域时，值被自动释放
- 编译器在编译时检查所有内存访问的合法性

这意味着：如果你的 Rust 代码能编译通过，它就不会有内存泄漏、悬垂指针、数据竞争等问题。

### Rust vs 其他语言

| 特性 | C/C++ | Go | Java | Rust |
|------|-------|-----|------|------|
| 内存管理 | 手动 | GC | GC | 所有权系统 |
| 内存安全 | 不保证 | 保证 | 保证 | 编译时保证 |
| 性能 | 极高 | 高 | 中高 | 极高 |
| 并发安全 | 不保证 | 部分保证 | 部分保证 | 编译时保证 |
| 零成本抽象 | 部分 | 否 | 否 | 是 |
| 学习曲线 | 高 | 低 | 中 | 高（但值得） |

## 2. Rust 的应用领域

- **系统编程**：操作系统（Redox OS）、驱动程序、嵌入式
- **命令行工具**：ripgrep、fd、bat、exa、starship（替代传统 Unix 工具）
- **Web 后端**：Actix-web、Axum（性能碾压大多数框架）
- **WebAssembly**：Rust 是 WASM 的一等公民
- **区块链**：Solana、Polkadot 核心用 Rust 编写
- **游戏引擎**：Bevy 引擎
- **基础设施**：Cloudflare Workers、Deno 运行时、SWC 编译器

## 3. 环境搭建

### 安装 Rust

Rust 通过 `rustup` 工具链管理器安装，它管理 Rust 编译器版本和相关工具。

```bash
# Windows：下载并运行 rustup-init.exe
# https://rustup.rs/

# Linux/macOS：
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh

# 验证安装
rustc --version    # Rust 编译器
cargo --version    # 包管理器和构建工具
rustup --version   # 工具链管理器
```

### 工具链说明

| 工具 | 作用 |
|------|------|
| `rustc` | Rust 编译器（通常不直接使用） |
| `cargo` | 包管理器 + 构建系统 + 测试运行器（核心工具） |
| `rustup` | 管理 Rust 版本和组件 |
| `rustfmt` | 代码格式化 |
| `clippy` | 代码 lint 工具（比编译器更严格的建议） |
| `rust-analyzer` | IDE 语言服务器（VS Code 必装） |

### VS Code 配置

推荐安装的扩展：
- **rust-analyzer**：代码补全、类型提示、跳转定义（必装）
- **Even Better TOML**：Cargo.toml 语法高亮
- **CodeLLDB**：调试支持

```json
// settings.json 推荐配置
{
  "rust-analyzer.check.command": "clippy",
  "rust-analyzer.inlayHints.typeHints.enable": true,
  "rust-analyzer.inlayHints.parameterHints.enable": true
}
```

### Cargo：Rust 的核心工具

Cargo 相当于 npm + webpack + jest 的合体，是 Rust 开发的核心。

```bash
# 创建新项目
cargo new my_project        # 创建二进制项目
cargo new my_lib --lib      # 创建库项目

# 构建和运行
cargo build                 # 编译（debug 模式）
cargo build --release       # 编译（release 模式，优化后）
cargo run                   # 编译并运行
cargo run --release         # 以 release 模式运行

# 检查和测试
cargo check                 # 只做类型检查，不生成二进制（比 build 快很多）
cargo test                  # 运行测试
cargo clippy                # 运行 lint 检查

# 依赖管理
cargo add serde             # 添加依赖
cargo update                # 更新依赖
```

**为什么 Cargo 如此重要？**

不同于 C/C++ 生态中混乱的构建系统（Makefile、CMake、Bazel...），Rust 从一开始就统一了包管理和构建系统。这意味着：
- 所有 Rust 项目结构一致，任何人都能立即上手
- 依赖管理简单可靠（类似 npm，但更严格）
- 交叉编译、测试、文档生成都是内置功能

## 4. 项目结构

```bash
cargo new hello_rust
```

生成的目录结构：

```
hello_rust/
├── Cargo.toml      # 项目配置文件（类似 package.json）
├── src/
│   └── main.rs     # 入口文件
└── .gitignore
```

### Cargo.toml

```toml
[package]
name = "hello_rust"
version = "0.1.0"
edition = "2021"        # Rust 版次（2015/2018/2021/2024）

[dependencies]
# 依赖在这里添加
# serde = "1.0"
# tokio = { version = "1", features = ["full"] }
```

**edition 是什么？**

Rust 每 3 年发布一个"版次"（edition），允许引入不向后兼容的语法改进，但不同 edition 的 crate 可以互相依赖。这是 Rust 在"稳定性"和"进化"之间的平衡设计。

## 5. 第一个 Rust 程序

### Hello World

```rust
fn main() {
    println!("Hello, World!");
}
```

逐行解析：
- `fn main()`：程序入口函数，`fn` 是函数关键字
- `println!`：这是一个**宏**（macro），不是函数。`!` 表示这是宏调用
- 字符串用双引号，语句以分号结尾

**为什么 println 是宏而不是函数？**

因为 `println!` 需要在编译时解析格式字符串，检查参数数量和类型是否匹配。普通函数做不到这一点。这是 Rust "零成本抽象"的体现——格式化检查在编译时完成，运行时没有额外开销。

### 稍微复杂的例子

```rust
use std::io;

fn main() {
    println!("请输入你的名字：");

    let mut name = String::new();

    io::stdin()
        .read_line(&mut name)
        .expect("读取输入失败");

    let name = name.trim(); // 去除换行符

    println!("你好，{}！欢迎学习 Rust。", name);
    println!("你的名字有 {} 个字符。", name.len());
}
```

这个例子涉及的概念（后续章节会详细讲解）：
- `let mut`：可变变量绑定
- `String::new()`：创建堆上的字符串
- `&mut name`：可变引用（借用）
- `.expect()`：错误处理（简单版）
- `let name = name.trim()`：变量遮蔽（shadowing）

## 6. 编译过程

```
源码 (.rs) → 编译器前端 (解析、类型检查、借用检查)
           → MIR (中间表示)
           → LLVM IR
           → 机器码 (可执行文件)
```

**Rust 编译为什么慢？**

Rust 编译器做了大量工作：
1. 类型推断和检查
2. 所有权和借用检查（borrow checker）
3. 单态化（monomorphization）：为每个泛型实例生成专用代码
4. LLVM 优化

这些工作换来的是：运行时零开销、无 GC、无运行时错误。编译慢是"把运行时的问题提前到编译时解决"的代价。

**加速编译的技巧：**
- 开发时用 `cargo check` 代替 `cargo build`
- 使用 `sccache` 缓存编译结果
- 使用 `mold`（Linux）或 `lld` 链接器
- 拆分大 crate 为多个小 crate

## 7. Rust 的版本发布模型

```
Nightly → Beta → Stable
  │                │
  │ 每6周          │ 每6周
  └────────────────┘
```

- **Stable**：生产环境使用，保证向后兼容
- **Beta**：下一个 stable 的候选版
- **Nightly**：最新特性，可能不稳定

大多数项目使用 stable 即可。只有少数场景需要 nightly（如某些宏特性、嵌入式开发）。

## 练习题

### 练习 1（入门）：项目初始化

创建一个 Cargo 项目 `rust_calculator`，在 `main.rs` 中实现一个简单的命令行计算器：
- 从标准输入读取两个数字和一个运算符（+、-、*、/）
- 输出计算结果
- 处理除以零的情况（打印错误信息）

### 练习 2（进阶）：多文件项目

在练习 1 的基础上：
- 将计算逻辑拆分到 `src/calculator.rs` 模块中
- `main.rs` 只负责输入输出
- 在 `Cargo.toml` 中添加 `colored` 依赖，让错误信息显示为红色

### 练习 3（扩展）：构建配置

- 添加一个 `build.rs` 构建脚本，在编译时打印当前日期
- 在程序启动时显示版本号（从 Cargo.toml 读取）
- 尝试用 `cargo build --release` 编译，对比 debug 和 release 的二进制大小

---

**思路提示：**
- 练习 1：使用 `io::stdin().read_line()` 读取输入，`str::parse::<f64>()` 解析数字，`match` 匹配运算符
- 练习 2：使用 `mod calculator;` 声明模块，`pub fn` 导出函数。`colored` crate 的用法：`"text".red()`
- 练习 3：`env!("CARGO_PKG_VERSION")` 宏可以在编译时获取版本号；`build.rs` 中用 `println!("cargo:warning=...")` 输出信息
