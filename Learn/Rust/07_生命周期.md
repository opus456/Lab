# 07 生命周期（Lifetimes） ⭐⭐核心难点

> 生命周期是 Rust 最独特的概念之一，它让编译器在编译期就能保证所有引用都是有效的，从而避免悬垂引用（dangling reference）。

---

## 1. 为什么需要生命周期

### 1.1 悬垂引用问题

```rust
fn main() {
    let r;                // ---------+-- 'a
    {                     //          |
        let x = 5;       // -+-- 'b  |
        r = &x;          //  |       |
    }                     // -+       |
    // println!("{}", r); // 错误！x 已被释放，r 是悬垂引用
}                         // ---------+
```

### 1.2 为什么这样设计

在 C/C++ 中，悬垂指针是最常见的内存安全 bug 之一。Rust 通过生命周期系统，在**编译期**就拒绝所有可能产生悬垂引用的代码。代价是需要偶尔写一些标注，但换来的是零运行时开销的内存安全。

### 1.3 核心直觉

> 生命周期回答一个问题：**这个引用能活多久？**
>
> 编译器需要确保：引用的生命周期 ≤ 被引用数据的生命周期。

## 2. 生命周期标注语法

### 2.1 基本语法

```rust
&i32        // 普通引用
&'a i32     // 带生命周期标注的引用
&'a mut i32 // 带生命周期标注的可变引用
```

### 2.2 重要认知

生命周期标注**不会改变**引用实际能活多久。它们只是告诉编译器多个引用之间的生命周期关系，帮助编译器做检查。

```rust
// 'a 不是"创造"一个生命周期，而是"描述"一个已存在的关系
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

## 3. 函数中的生命周期

### 3.1 何时需要标注

当函数**接收引用并返回引用**时，编译器需要知道返回的引用与哪个输入参数的生命周期相关联。

```rust
// 编译器无法推断返回值的生命周期来自 x 还是 y
// 所以必须显式标注
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}

fn main() {
    let string1 = String::from("长字符串");
    let result;
    {
        let string2 = String::from("短");
        result = longest(string1.as_str(), string2.as_str());
        println!("更长的是: {}", result); // OK，string2 还活着
    }
    // println!("{}", result); // 错误！string2 已被释放
}
```

### 3.2 只关联部分参数

```rust
// 返回值只与 x 的生命周期相关
fn first_word<'a>(x: &'a str, _y: &str) -> &'a str {
    x.split_whitespace().next().unwrap_or("")
}
```

### 3.3 返回值不能引用局部变量

```rust
// 错误！不能返回对局部变量的引用
fn invalid<'a>() -> &'a str {
    let s = String::from("hello");
    // &s  // s 在函数结束时被释放，引用会悬垂
    "hello" // 字符串字面量有 'static 生命周期，这是 OK 的
}
```

## 4. 结构体中的生命周期

### 4.1 持有引用的结构体

当结构体持有引用时，必须标注生命周期，确保结构体不会比它引用的数据活得更久。

```rust
// ImportantExcerpt 不能比 part 引用的数据活得更久
struct ImportantExcerpt<'a> {
    part: &'a str,
}

fn main() {
    let novel = String::from("Call me Ishmael. Some years ago...");
    let first_sentence;
    {
        first_sentence = novel.split('.').next().expect("找不到'.'");
    }
    // OK: novel 比 i 活得久
    let i = ImportantExcerpt { part: first_sentence };
    println!("{}", i.part);
}
```

### 4.2 多个生命周期参数

```rust
struct TextWindow<'text, 'query> {
    content: &'text str,
    highlight: &'query str,
}

// 当两个引用来源不同且生命周期不同时，使用多个生命周期参数
impl<'text, 'query> TextWindow<'text, 'query> {
    fn contains_highlight(&self) -> bool {
        self.content.contains(self.highlight)
    }
}
```

### 4.3 为什么这样设计

结构体的生命周期标注让编译器能追踪"这个结构体里的引用指向哪里"。没有这个信息，编译器无法保证结构体在使用时其内部引用仍然有效。

## 5. 生命周期省略规则（Elision Rules）

### 5.1 好消息

Rust 编译器有三条省略规则，能自动推断大多数情况下的生命周期。这意味着**实际编码中需要手写生命周期标注的场景比想象中少得多**。

### 5.2 三条规则

**规则一：** 每个引用参数都获得自己的生命周期参数。

```rust
fn foo(x: &str, y: &str)
// 等价于
fn foo<'a, 'b>(x: &'a str, y: &'b str)
```

**规则二：** 如果只有一个输入生命周期参数，它被赋给所有输出生命周期。

```rust
fn first_word(s: &str) -> &str
// 等价于
fn first_word<'a>(s: &'a str) -> &'a str
```

**规则三：** 如果有多个输入生命周期参数，但其中一个是 `&self` 或 `&mut self`，则 `self` 的生命周期被赋给所有输出。

```rust
impl MyStruct {
    fn get_part(&self, input: &str) -> &str
    // 等价于
    fn get_part<'a, 'b>(&'a self, input: &'b str) -> &'a str
}
```

### 5.3 何时规则不够用

```rust
// 两个输入引用，没有 self —— 编译器不知道返回值跟谁关联
fn longest(x: &str, y: &str) -> &str // 编译错误！
// 必须手动标注
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str // OK
```

## 6. 'static 生命周期

### 6.1 含义

`'static` 表示引用在**整个程序运行期间**都有效。

```rust
// 字符串字面量都是 'static
let s: &'static str = "我会活到程序结束";

// 泄漏的内存也是 'static（不推荐）
let leaked: &'static str = Box::leak(Box::new(String::from("泄漏")));
```

### 6.2 常见误区

```rust
// T: 'static 不意味着 T 必须是引用！
// 它意味着 T 不包含任何非 'static 的引用
fn spawn_thread<T: Send + 'static>(val: T) {
    std::thread::spawn(move || {
        println!("线程中使用值");
    });
}

// String 满足 'static（它拥有自己的数据，不含引用）
// &'a str 只有当 'a = 'static 时才满足
```

### 6.3 何时使用 'static

- 线程间传递数据时（`thread::spawn` 要求 `'static`）
- 全局常量
- 错误消息字符串
- 作为 trait bound 表示"不含短生命周期引用"

## 7. 泛型上的生命周期约束

### 7.1 生命周期作为泛型约束

```rust
use std::fmt::Display;

fn longest_with_announcement<'a, T>(
    x: &'a str,
    y: &'a str,
    ann: T,
) -> &'a str
where
    T: Display,
{
    println!("公告: {}", ann);
    if x.len() > y.len() { x } else { y }
}
```

### 7.2 生命周期子类型（Subtyping）

```rust
// 'long: 'short 表示 'long 至少和 'short 一样长
fn select<'long: 'short, 'short>(
    long_lived: &'long str,
    _short_lived: &'short str,
) -> &'short str {
    long_lived // OK: 'long 比 'short 活得久，可以"缩短"
}
```

### 7.3 结构体上的生命周期约束

```rust
struct Wrapper<'a, T: 'a> {
    value: &'a T,
}
// T: 'a 表示 T 中的任何引用都至少活 'a 那么久
// 这保证了 value 引用的 T 不会先于 Wrapper 被释放
```

## 8. 常见生命周期模式

### 8.1 迭代器返回引用

```rust
struct StrSplit<'haystack, 'delimiter> {
    remainder: Option<&'haystack str>,
    delimiter: &'delimiter str,
}

impl<'haystack, 'delimiter> Iterator for StrSplit<'haystack, 'delimiter> {
    type Item = &'haystack str;

    fn next(&mut self) -> Option<Self::Item> {
        let remainder = self.remainder.as_mut()?;
        if let Some(next_delim) = remainder.find(self.delimiter) {
            let until_delimiter = &remainder[..next_delim];
            *remainder = &remainder[(next_delim + self.delimiter.len())..];
            Some(until_delimiter)
        } else {
            self.remainder.take()
        }
    }
}
```

### 8.2 缓存模式

```rust
use std::collections::HashMap;

struct Cache {
    data: HashMap<String, String>,
}

impl Cache {
    fn new() -> Self {
        Cache { data: HashMap::new() }
    }

    // 返回对内部数据的引用，生命周期与 &self 绑定
    fn get_or_insert(&mut self, key: &str, value: String) -> &str {
        self.data.entry(key.to_string()).or_insert(value)
    }
}
```

### 8.3 Builder 模式中的引用

```rust
struct RequestBuilder<'a> {
    url: &'a str,
    headers: Vec<(&'a str, &'a str)>,
}

impl<'a> RequestBuilder<'a> {
    fn new(url: &'a str) -> Self {
        RequestBuilder { url, headers: vec![] }
    }

    fn header(mut self, key: &'a str, value: &'a str) -> Self {
        self.headers.push((key, value));
        self
    }

    fn build(&self) -> String {
        format!("GET {} with {} headers", self.url, self.headers.len())
    }
}
```

## 9. 实战技巧

### 9.1 何时该用引用 vs 拥有所有权

| 场景 | 选择 | 原因 |
|------|------|------|
| 结构体需要长期持有数据 | `String` / `Vec<T>` | 避免生命周期复杂性 |
| 只是临时读取 | `&str` / `&[T]` | 零拷贝，高性能 |
| 跨线程传递 | 拥有所有权 | `'static` 约束 |
| 解析器/迭代器 | 引用 | 避免分配，指向源数据 |

### 9.2 常见错误与解决方案

```rust
// 错误：返回对局部变量的引用
fn bad() -> &str {
    let s = String::from("hello");
    &s // 错误！
}
// 解决：返回 String（拥有所有权）
fn good() -> String {
    String::from("hello")
}

// 错误：结构体比引用的数据活得久
struct Bad<'a> { data: &'a str }
fn create_bad() -> Bad<'static> {
    let s = String::from("temp");
    // Bad { data: &s } // 错误！
    Bad { data: "static string" } // OK
}
```

### 9.3 心智模型

1. **先尝试不写生命周期** —— 省略规则能处理大多数情况
2. **编译器报错时再加** —— 错误信息通常会告诉你该加在哪
3. **如果生命周期太复杂** —— 考虑改用拥有所有权的类型（`String` 代替 `&str`）
4. **`clone()` 不是罪** —— 正确性优先，性能后优化

## 总结

| 概念 | 核心要点 |
|------|----------|
| 生命周期标注 | 描述引用间关系，不改变实际生命周期 |
| 省略规则 | 三条规则覆盖大多数场景 |
| 结构体生命周期 | 确保结构体不比其引用的数据活得久 |
| `'static` | 整个程序期间有效，或不含短引用 |
| 实用建议 | 先不写，编译器报错再加，太复杂就用所有权 |
