# 05 错误处理 ⭐重点

> 难度：★★★☆☆
>
> Rust 没有异常机制。它用类型系统来表达错误，让编译器强制你处理每一个可能的失败。
> 这看起来"啰嗦"，但实际上消除了一整类运行时 bug。

---

## 1. panic 与不可恢复错误

### 1.1 什么时候该 panic

```rust
fn main() {
    // panic! 用于不可恢复的错误——程序的逻辑前提被违反
    // 比如：索引越界、断言失败、不应该到达的代码路径

    let v = vec![1, 2, 3];
    // v[99]; // 这会 panic: index out of bounds

    // 显式 panic
    // panic!("出了大问题！");

    // unwrap/expect 本质上也是 panic
    let _num: i32 = "not_a_number".parse().unwrap(); // panic!
}
```

### 1.2 panic 的行为

```rust
// 默认行为：展开（unwind）调用栈，逐层清理
// 可以在 Cargo.toml 中配置为直接终止（abort）：
// [profile.release]
// panic = 'abort'

// catch_unwind 可以捕获 panic（类似 try-catch，但不推荐常规使用）
use std::panic;

fn main() {
    let result = panic::catch_unwind(|| {
        panic!("测试 panic");
    });

    match result {
        Ok(_) => println!("没有 panic"),
        Err(_) => println!("捕获到 panic"),
    }
    println!("程序继续运行");
}
```

**为什么这样设计：**
- panic 用于"程序员的错误"（bug），不是"用户的错误"（输入问题）
- 展开栈保证了 RAII 资源的正确释放（文件关闭、锁释放等）
- 与 Result 的分工明确：可预期的失败用 Result，不可预期的 bug 用 panic

## 2. Result<T, E> 与可恢复错误

### 2.1 Result 基础

```rust
use std::fs::File;
use std::io::Read;

fn read_file_content(path: &str) -> Result<String, std::io::Error> {
    let mut file = File::open(path)?;  // ? 操作符：出错就提前返回
    let mut content = String::new();
    file.read_to_string(&mut content)?;
    Ok(content)
}
fn main() {
    match read_file_content("hello.txt") {
        Ok(content) => println!("文件内容: {}", content),
        Err(e) => println!("读取失败: {}", e),
    }

    // 也可以用 unwrap_or_else 提供默认行为
    let content = read_file_content("config.toml")
        .unwrap_or_else(|_| String::from("default config"));
    println!("{}", content);
}
```

### 2.2 Result 的常用方法

```rust
fn main() {
    let good: Result<i32, &str> = Ok(42);
    let bad: Result<i32, &str> = Err("出错了");

    // map：转换 Ok 中的值
    let doubled = good.map(|x| x * 2);  // Ok(84)

    // and_then：链式操作（类似 flatMap）
    let result = good.and_then(|x| {
        if x > 0 { Ok(x) } else { Err("必须为正数") }
    });

    // unwrap_or：提供默认值
    let val = bad.unwrap_or(0);  // 0

    // map_err：转换错误类型
    let mapped: Result<i32, String> = bad.map_err(|e| format!("错误: {}", e));

    // is_ok / is_err
    assert!(good.is_ok());
    assert!(bad.is_err());

    // ok() 转换为 Option<T>
    let opt: Option<i32> = good.ok();  // Some(42)
}
```

## 3. Option<T>

### 3.1 Option 基础

```rust
fn find_user(id: u32) -> Option<String> {
    match id {
        1 => Some(String::from("Alice")),
        2 => Some(String::from("Bob")),
        _ => None,
    }
}

fn main() {
    // 模式匹配
    match find_user(1) {
        Some(name) => println!("找到用户: {}", name),
        None => println!("用户不存在"),
    }

    // if let（只关心 Some 的情况）
    if let Some(name) = find_user(2) {
        println!("Hello, {}!", name);
    }

    // 链式操作
    let upper_name = find_user(1)
        .map(|name| name.to_uppercase())      // Some("ALICE")
        .unwrap_or_else(|| "UNKNOWN".into()); // 如果 None 则用默认值

    println!("{}", upper_name);

    // Option 与 Result 的转换
    let opt: Option<i32> = Some(42);
    let res: Result<i32, &str> = opt.ok_or("值不存在");  // Ok(42)

    let none: Option<i32> = None;
    let res2: Result<i32, &str> = none.ok_or("值不存在"); // Err("值不存在")
}
```

**为什么这样设计：**
- Option 替代了 null/nil——编译器强制你处理"值可能不存在"的情况
- 不会出现 NullPointerException，因为你必须显式解包
- `?` 操作符让 Option 和 Result 的传播一样简洁

## 4. ? 操作符详解

### 4.1 基本用法

```rust
use std::fs;
use std::io;

// ? 操作符：如果是 Err，提前返回；如果是 Ok，解包继续
fn read_username_from_file() -> Result<String, io::Error> {
    let mut username = fs::read_to_string("username.txt")?;
    username.trim_end().to_string();
    Ok(username)
}

// ? 也可以用于 Option（在返回 Option 的函数中）
fn first_even(numbers: &[i32]) -> Option<i32> {
    let first = numbers.first()?;  // 如果为空，返回 None
    if first % 2 == 0 {
        Some(*first)
    } else {
        None
    }
}

// 在 main 中使用 ?
fn main() -> Result<(), Box<dyn std::error::Error>> {
    let content = fs::read_to_string("config.txt")?;
    let num: i32 = content.trim().parse()?;
    println!("配置值: {}", num);
    Ok(())
}
```

### 4.2 ? 操作符的错误转换

```rust
use std::num::ParseIntError;
use std::io;

#[derive(Debug)]
enum AppError {
    Io(io::Error),
    Parse(ParseIntError),
}

// 实现 From trait 让 ? 自动转换错误类型
impl From<io::Error> for AppError {
    fn from(e: io::Error) -> Self {
        AppError::Io(e)
    }
}

impl From<ParseIntError> for AppError {
    fn from(e: ParseIntError) -> Self {
        AppError::Parse(e)
    }
}

// 现在 ? 可以自动将 io::Error 和 ParseIntError 转换为 AppError
fn read_number_from_file(path: &str) -> Result<i32, AppError> {
    let content = std::fs::read_to_string(path)?;  // io::Error -> AppError
    let number = content.trim().parse::<i32>()?;   // ParseIntError -> AppError
    Ok(number)
}
```

**为什么这样设计：**
- `?` 是语法糖，等价于 `match result { Ok(v) => v, Err(e) => return Err(e.into()) }`
- `From` trait 的自动转换让不同错误类型可以统一处理
- 比 try-catch 更显式——你能清楚看到哪些操作可能失败

## 5. 自定义错误类型

### 5.1 手动实现

```rust
use std::fmt;
use std::error::Error;

#[derive(Debug)]
enum DatabaseError {
    ConnectionFailed(String),
    QueryFailed { query: String, reason: String },
    NotFound(String),
    Timeout(u64),
}

impl fmt::Display for DatabaseError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            Self::ConnectionFailed(url) =>
                write!(f, "无法连接到数据库: {}", url),
            Self::QueryFailed { query, reason } =>
                write!(f, "查询失败 '{}': {}", query, reason),
            Self::NotFound(item) =>
                write!(f, "未找到: {}", item),
            Self::Timeout(ms) =>
                write!(f, "操作超时 ({}ms)", ms),
        }
    }
}

impl Error for DatabaseError {}
```
