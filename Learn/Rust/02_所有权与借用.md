# 第二章：所有权与借用

> **难度：⭐ 重点难点** | 这是 Rust 最核心的概念。理解所有权系统，就理解了 Rust 为什么能在没有 GC 的情况下保证内存安全。本章需要反复阅读和练习。

---

## 1. 内存管理的三种方式

在理解所有权之前，先看看其他语言如何管理内存：

| 方式 | 语言 | 优点 | 缺点 |
|------|------|------|------|
| 手动管理 | C/C++ | 性能最好 | 容易出错（内存泄漏、悬垂指针） |
| 垃圾回收 (GC) | Java/Go/JS | 安全简单 | GC 停顿、内存开销大 |
| 所有权系统 | Rust | 安全 + 高性能 | 学习曲线陡峭 |

Rust 的所有权系统在编译时检查内存使用规则，违反规则的代码无法编译。这意味着：
- 零运行时开销（没有 GC）
- 不可能出现内存安全问题（编译器保证）

## 2. 所有权规则

Rust 的所有权有三条核心规则：

1. **每个值都有一个所有者（owner）**
2. **同一时刻只能有一个所有者**
3. **当所有者离开作用域，值被自动释放（drop）**

```rust
fn main() {
    {
        let s = String::from("hello"); // s 进入作用域，成为 "hello" 的所有者
        println!("{}", s);             // s 有效，可以使用
    } // s 离开作用域，String 的内存被自动释放（调用 drop）

    // println!("{}", s); // Error: s 已经不存在了
}
```

**为什么这样设计？**

这解决了 C/C++ 中最常见的两个问题：
- **内存泄漏**：忘记 free → Rust 自动在作用域结束时释放
- **悬垂指针**：free 后继续使用 → Rust 不允许在所有者失效后访问

## 3. 移动语义 (Move)

### 栈上数据：复制

```rust
fn main() {
    let x = 5;
    let y = x;  // 复制！x 和 y 都有效
    println!("x={}, y={}", x, y); // OK
}
```

基本类型（整数、浮点、布尔、字符、纯栈元组）实现了 `Copy` trait，赋值时自动复制。因为它们大小固定且在栈上，复制成本极低。

### 堆上数据：移动

```rust
fn main() {
    let s1 = String::from("hello");
    let s2 = s1;  // 移动！s1 的所有权转移给 s2

    // println!("{}", s1); // Error: value used here after move
    println!("{}", s2);    // OK
}
```

**为什么 String 是移动而不是复制？**

`String` 在堆上分配内存。如果简单复制指针（浅拷贝），两个变量指向同一块内存，作用域结束时会 double free。如果深拷贝，性能开销大且不可预测。

Rust 的选择：**移动**——转移所有权，旧变量失效。这既安全又高效（只复制栈上的指针/长度/容量，不复制堆数据）。

```
栈内存：                堆内存：
s1: [ptr, len, cap]    "hello"
     ↓ 移动后
s1: (无效)
s2: [ptr, len, cap] → "hello"
```

### 函数调用中的移动

```rust
fn take_ownership(s: String) {
    println!("I own: {}", s);
} // s 在这里被释放

fn main() {
    let s = String::from("hello");
    take_ownership(s);  // s 的所有权移动到函数参数

    // println!("{}", s); // Error: s 已经被移动了
}
```

```rust
fn give_ownership() -> String {
    let s = String::from("hello");
    s  // 返回值将所有权移动给调用者
}

fn main() {
    let s = give_ownership(); // 接收所有权
    println!("{}", s);        // OK
}
```

## 4. 克隆 (Clone)

如果你确实需要深拷贝，使用 `clone()`：

```rust
fn main() {
    let s1 = String::from("hello");
    let s2 = s1.clone();  // 深拷贝堆数据

    println!("s1={}, s2={}", s1, s2); // 两个都有效
}
```

`clone()` 是显式的，让你知道这里有性能开销。Rust 不会偷偷帮你做昂贵的操作。

## 5. 引用与借用 (References & Borrowing)

每次传递所有权太麻烦了。大多数时候你只想"借用"一下数据，用完还回去。

### 不可变引用 &T

```rust
fn calculate_length(s: &String) -> usize {
    s.len()
    // s 是借用的，函数结束时不会释放原始数据
}

fn main() {
    let s = String::from("hello");
    let len = calculate_length(&s);  // & 创建引用，借出 s
    println!("'{}' has length {}", s, len); // s 仍然有效！
}
```

**借用规则：**
- 引用不拥有数据，只是"借来看看"
- 可以同时存在多个不可变引用（多人同时读没问题）
- 引用必须始终有效（不能指向已释放的数据）

### 可变引用 &mut T

```rust
fn append_world(s: &mut String) {
    s.push_str(", world!");
}

fn main() {
    let mut s = String::from("hello");
    append_world(&mut s);  // 可变借用
    println!("{}", s);     // "hello, world!"
}
```

### 借用的核心限制

```rust
fn main() {
    let mut s = String::from("hello");

    // 规则：同一时刻，要么有一个可变引用，要么有多个不可变引用
    let r1 = &s;     // OK：不可变引用
    let r2 = &s;     // OK：多个不可变引用
    println!("{}, {}", r1, r2);

    let r3 = &mut s; // OK：r1 和 r2 已经不再使用（NLL）
    r3.push_str("!");
    println!("{}", r3);

    // 但不能同时存在可变和不可变引用：
    // let r4 = &s;
    // let r5 = &mut s;
    // println!("{}, {}", r4, r5); // Error!
}
```

**为什么有这个限制？**

这是为了防止**数据竞争**（data race）。数据竞争发生在：
1. 两个指针同时访问同一数据
2. 至少一个在写入
3. 没有同步机制

Rust 在编译时就消除了数据竞争的可能性。这个规则看起来严格，但它保证了：
- 多个读者同时读 → 安全
- 一个写者独占写 → 安全
- 读者和写者同时存在 → 不允许（编译错误）

### NLL (Non-Lexical Lifetimes)

```rust
fn main() {
    let mut s = String::from("hello");

    let r1 = &s;
    let r2 = &s;
    println!("{} and {}", r1, r2);
    // r1 和 r2 在这之后不再使用，它们的生命周期在这里结束

    let r3 = &mut s; // OK！因为 r1、r2 已经"死了"
    println!("{}", r3);
}
```

Rust 编译器足够聪明，引用的生命周期不是到作用域结束，而是到最后一次使用。

## 6. 悬垂引用 (Dangling References)

```rust
// 这段代码无法编译！
fn dangle() -> &String {
    let s = String::from("hello");
    &s  // 返回对局部变量的引用
} // s 在这里被释放，引用指向无效内存！

// 正确做法：返回所有权
fn no_dangle() -> String {
    let s = String::from("hello");
    s  // 移动所有权给调用者
}
```

编译器保证：引用永远不会指向无效数据。这在 C/C++ 中是最常见的 bug 来源之一。

## 7. 切片 (Slice)

切片是对集合中一段连续元素的引用，不拥有数据。

```rust
fn first_word(s: &str) -> &str {
    let bytes = s.as_bytes();
    for (i, &byte) in bytes.iter().enumerate() {
        if byte == b' ' {
            return &s[..i];
        }
    }
    &s[..]
}

fn main() {
    let s = String::from("hello world");
    let word = first_word(&s);
    println!("first word: {}", word); // "hello"

    // s.clear(); // Error! 不能在有不可变借用时修改 s
    // 这就是借用检查器保护你的方式
}
```

**&str vs String：**
- `String`：拥有数据的堆分配字符串，可增长
- `&str`：字符串切片，是对某段 UTF-8 数据的引用
- 函数参数优先用 `&str`（更通用，String 和字面量都能传入）

```rust
// 好的实践：参数用 &str
fn greet(name: &str) {
    println!("Hello, {}!", name);
}

fn main() {
    let owned = String::from("Alice");
    greet(&owned);    // String → &str（自动解引用）
    greet("Bob");     // &str 字面量直接传入
}
```

## 8. 所有权与实际工程

### 常见模式

```rust
// 模式 1：函数借用数据，不获取所有权
fn process(data: &[u8]) -> usize {
    data.len()
}

// 模式 2：函数需要拥有数据（如存入结构体）
struct Cache {
    data: String,
}

impl Cache {
    fn new(data: String) -> Self {
        Cache { data } // 获取所有权，存入结构体
    }
}

// 模式 3：返回新创建的数据
fn create_greeting(name: &str) -> String {
    format!("Hello, {}!", name) // 创建新 String 并返回所有权
}
```

### 所有权思维方式

问自己这些问题：
1. 这个数据谁负责释放？→ 所有者
2. 我只是临时用一下还是要长期持有？→ 借用 vs 所有权
3. 我需要修改数据吗？→ `&mut` vs `&`
4. 多个地方需要同时访问吗？→ 多个 `&` 或者 `Rc`/`Arc`（后续章节）

## 练习题

### 练习 1（基础）：所有权转移

编写一个函数 `longest_string`，接受两个 `String` 参数，返回较长的那个（获取所有权）。思考：另一个 String 会怎样？

然后改写为借用版本 `longest_str`，接受两个 `&str`，返回 `&str`。思考：返回的引用的生命周期应该是什么？

### 练习 2（融合）：文本处理器

实现一个简单的文本处理器：
- 函数 `count_words(text: &str) -> usize` 统计单词数
- 函数 `extract_emails(text: &str) -> Vec<&str>` 提取所有邮箱地址
- 函数 `replace_word(text: &mut String, old: &str, new_word: &str)` 替换单词
- 在 main 中组合使用这些函数，注意借用规则

### 练习 3（工程实践）：简易日志系统

设计一个日志系统：
- `struct LogEntry` 包含时间戳、级别、消息
- `struct Logger` 持有一个 `Vec<LogEntry>`
- 实现 `log(&mut self, level: &str, message: String)` 方法
- 实现 `search(&self, keyword: &str) -> Vec<&LogEntry>` 方法
- 思考：为什么 message 用 String（获取所有权）而 keyword 用 &str（借用）？

### 练习 4（挑战）：内存池

实现一个简单的字符串内存池：
- `struct StringPool` 内部用 `Vec<String>` 存储字符串
- `intern(&mut self, s: &str) -> &str`：如果字符串已存在则返回引用，否则存入并返回引用
- 思考：这里的生命周期关系是什么？为什么这个设计在安全 Rust 中很难实现？

---

**思路提示：**
- 练习 1：借用版本需要生命周期标注 `fn longest_str<'a>(s1: &'a str, s2: &'a str) -> &'a str`，因为编译器需要知道返回的引用活多久
- 练习 2：注意 `replace_word` 需要 `&mut String`，调用它时不能同时持有不可变引用。可能需要调整调用顺序
- 练习 3：`message` 用 String 是因为 Logger 需要拥有这个数据（存储起来）；`keyword` 用 &str 是因为搜索只需要临时借用
- 练习 4：这涉及自引用结构的问题。安全 Rust 中可以用索引代替引用，或者使用 `unsafe`。这是一个经典的 Rust 难题
