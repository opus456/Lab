# 09 模块与包管理

> **难度：中等** | Rust 工程不是单文件脚本，模块、crate 和 Cargo workspace 决定了项目能否长期维护。

---

## 1. crate、package 与 module

**标注：核心高频**

Rust 的代码组织有三层：

| 概念 | 含义 | 常见文件 |
|------|------|----------|
| package | 一个 Cargo 项目 | `Cargo.toml` |
| crate | 一个编译单元 | `src/main.rs` 或 `src/lib.rs` |
| module | crate 内部的命名空间 | `mod xxx` |

```text
my_app/
├── Cargo.toml
└── src/
    ├── main.rs
    ├── config.rs
    └── service/
        ├── mod.rs
        └── user.rs
```

```rust
mod config;
mod service;

fn main() {
    let value = config::load();
    println!("{}", value);
}
```

**为什么这样设计、这样设计带来的好处：**

Rust 把“构建边界”和“命名空间边界”分开。crate 是编译和依赖边界，module 是组织代码的边界，这让大型项目既能拆分编译单元，也能保持内部结构清晰。

## 2. 可见性：默认私有

**标注：核心高频**

Rust 中模块项默认私有，只有显式 `pub` 才能被外部访问。

```rust
mod user {
    pub struct User {
        pub id: u64,
        name: String,
    }

    impl User {
        pub fn new(id: u64, name: String) -> Self {
            Self { id, name }
        }

        pub fn name(&self) -> &str {
            &self.name
        }
    }
}

fn main() {
    let user = user::User::new(1, "Alice".to_string());
    println!("{} {}", user.id, user.name());
}
```

**为什么这样设计、这样设计带来的好处：**

默认私有能保护模块内部不被外部随意依赖。工程中只暴露稳定 API，内部实现可自由重构。

## 3. `use` 与重导出

**标注：重要**

`use` 用来缩短路径，`pub use` 用来建立更稳定的公共 API。

```rust
mod domain {
    pub struct User {
        pub id: u64,
    }
}

pub use domain::User;

fn main() {
    let user = User { id: 1 };
    println!("{}", user.id);
}
```

**为什么这样设计、这样设计带来的好处：**

内部路径可能频繁调整，但外部用户只依赖 `crate::User` 这样的稳定入口。重导出就是为库设计 API 门面。

## 4. Cargo 依赖与 feature

**标注：核心高频**

Cargo 使用 `Cargo.toml` 管理依赖。

```toml
[dependencies]
serde = { version = "1", features = ["derive"] }
tokio = { version = "1", features = ["rt-multi-thread", "macros"] }
```

feature 允许按需启用能力，减少编译时间和二进制体积。

**为什么这样设计、这样设计带来的好处：**

Rust 生态强调零成本和可裁剪。feature 让库既能提供丰富能力，又不强迫所有用户承担全部依赖成本。

---

## 5. Workspace：多 crate 工程

**标注：重要**

当项目拆成多个 crate 时，可以使用 workspace 统一管理。

```toml
[workspace]
members = [
    "crates/api",
    "crates/core",
    "crates/cli"
]
resolver = "2"
```

```text
project/
├── Cargo.toml
└── crates/
    ├── core/
    ├── api/
    └── cli/
```

**为什么这样设计、这样设计带来的好处：**

Workspace 共享锁文件和构建缓存，适合把核心逻辑、命令行入口、Web API 拆成独立 crate，降低模块耦合。

## 动手练习

### 练习 1：拆分配置模块

把读取配置的逻辑放到 `config.rs`，主函数只调用 `config::load()`。

**思路提示：** 使用 `mod config;`，需要暴露的函数加 `pub`。

### 练习 2：设计领域模型 API

创建 `domain/user.rs`，字段私有，通过构造函数和 getter 访问。

**思路提示：** 用私有字段保护不变量，例如邮箱不能为空。

### 练习 3：库 crate 的重导出

在 `lib.rs` 中使用 `pub use` 暴露核心结构体。

**思路提示：** 外部调用方不应该关心内部目录层级。

### 练习 4：拆分 workspace

把 CLI 入口和核心业务拆成两个 crate。

**思路提示：** `cli` 依赖 `core`，业务逻辑不要写在 `main.rs`。
