# 18 综合实战项目：日志分析 CLI 工具

> **难度：实战** | 本章把前面学过的模块化、错误处理、集合、并发、CLI、日志与测试串成一个真实可用工具。

---

## 1. 项目目标

**标注：核心高频**

构建一个 `loglens` 命令行工具，用于分析本地日志文件：

- 统计不同级别日志数量
- 按关键词过滤日志
- 输出 JSON 或表格
- 支持大文件流式读取
- 提供基础测试和文档

```bash
loglens analyze app.log --level error --json
loglens grep app.log --keyword timeout
```

**为什么这样设计、这样设计带来的好处：**

日志分析是开发者日常真实需求。它既不会过度依赖复杂外部系统，又能覆盖 Rust 工程化核心能力。

## 2. 技术选型

**标注：重要**

| 需求 | 推荐库 | 理由 |
|------|--------|------|
| CLI 参数 | `clap` | derive API 成熟，帮助信息自动生成 |
| 序列化 | `serde` / `serde_json` | Rust 生态事实标准 |
| 错误处理 | `thiserror` / `anyhow` | 区分库错误与应用错误 |
| 日志 | `tracing` | 结构化日志，适合排错 |
| 测试 CLI | `assert_cmd` | 可测试命令行行为 |

```toml
[dependencies]
clap = { version = "4", features = ["derive"] }
serde = { version = "1", features = ["derive"] }
serde_json = "1"
thiserror = "1"
tracing = "0.1"
tracing-subscriber = "0.3"
```

**为什么这样设计、这样设计带来的好处：**

选型优先选择生态稳定、文档完善、维护活跃的库，降低学习项目迁移到真实项目时的成本。

## 3. 目录结构

**标注：核心高频**

```text
loglens/
├── Cargo.toml
├── README.md
├── src/
│   ├── main.rs
│   ├── lib.rs
│   ├── cli.rs
│   ├── error.rs
│   ├── parser.rs
│   ├── analyzer.rs
│   └── output.rs
└── tests/
    └── cli.rs
```

职责划分：

| 模块 | 职责 |
|------|------|
| `cli` | 参数定义与子命令 |
| `parser` | 日志行解析 |
| `analyzer` | 统计与过滤 |
| `output` | 表格或 JSON 输出 |
| `error` | 统一错误类型 |

**为什么这样设计、这样设计带来的好处：**

清晰分层能避免 CLI 参数、文件读取、业务统计混在一起。后续想支持目录递归、压缩日志、Web API 都更容易扩展。

## 4. 分阶段实现指引

**标注：核心高频**

### 阶段一：最小可用版本

实现读取单个文件并统计 `ERROR`、`WARN`、`INFO` 数量。

```rust
#[derive(Debug, Default, serde::Serialize)]
pub struct Summary {
    pub error: usize,
    pub warn: usize,
    pub info: usize,
    pub other: usize,
}
```

### 阶段二：CLI 子命令

使用 `clap` 定义 `analyze` 和 `grep` 子命令。

### 阶段三：流式读取

使用 `BufRead::lines()` 避免一次性读入大文件。

### 阶段四：输出格式

支持默认文本输出和 `--json` 输出。

### 阶段五：测试与文档

为解析函数写单元测试，为 CLI 写集成测试。

**为什么这样设计、这样设计带来的好处：**

分阶段实现能让每一步都有可运行成果，降低复杂项目的心理负担，也符合真实团队迭代方式。

## 5. 错误处理与日志策略

**标注：重要**

错误分两层：

- **用户可见错误**：文件不存在、参数不合法
- **开发者排查信息**：处理耗时、读取行数、过滤条件

```rust
use thiserror::Error;

#[derive(Debug, Error)]
pub enum AppError {
    #[error("failed to read file: {0}")]
    Io(#[from] std::io::Error),
    #[error("invalid log level: {0}")]
    InvalidLevel(String),
}
```

**为什么这样设计、这样设计带来的好处：**

用户需要清晰错误提示，开发者需要足够上下文定位问题。两者混在一起会让 CLI 难用、日志难查。

## 动手练习

### 练习 1：实现日志级别统计

读取一个日志文件，统计 `ERROR`、`WARN`、`INFO` 和其他行数。

**思路提示：** 先写纯函数处理 `&str`，再接文件读取。

### 练习 2：增加 CLI 参数

添加 `analyze <path> --json` 参数。

**思路提示：** 用 `clap` derive 定义结构体，主函数只负责分发。

### 练习 3：关键词过滤

实现 `grep <path> --keyword <text>`，输出匹配行。

**思路提示：** 用迭代器逐行处理，避免读取整个文件。

### 练习 4：补测试

为日志解析、统计、JSON 输出各写测试。

**思路提示：** 业务逻辑测试不要依赖真实文件，CLI 集成测试再使用临时文件。

### 练习 5：增加并发目录扫描

递归扫描目录下多个 `.log` 文件并汇总结果。

**思路提示：** 可以先用线程池或 rayon，注意错误汇总方式。
