# 15 工程化实践

> **难度：中等** | 工程化决定 Rust 项目能否被团队维护、测试、发布和长期演进。

---

## 1. 项目结构分层

**标注：核心高频**

真实项目不要把所有逻辑写在 `main.rs`。推荐把入口、领域逻辑、基础设施拆开。

```text
my_tool/
├── Cargo.toml
├── src/
│   ├── main.rs
│   ├── lib.rs
│   ├── config.rs
│   ├── error.rs
│   └── service.rs
└── tests/
    └── cli_test.rs
```

```rust
// src/lib.rs
pub mod config;
pub mod error;
pub mod service;
```

**为什么这样设计、这样设计带来的好处：**

`main.rs` 只负责组装依赖和启动流程，核心逻辑放进 `lib.rs` 后更容易测试和复用。

## 2. 统一错误类型

**标注：核心高频**

工程项目应避免到处 `unwrap`，推荐定义统一错误类型。

```toml
[dependencies]
thiserror = "1"
```

```rust
use thiserror::Error;

#[derive(Debug, Error)]
pub enum AppError {
    #[error("io error: {0}")]
    Io(#[from] std::io::Error),
    #[error("invalid config: {0}")]
    Config(String),
}

pub type Result<T> = std::result::Result<T, AppError>;
```

**为什么这样设计、这样设计带来的好处：**

统一错误类型让调用链可读，日志可聚合，CLI 或 Web 层可以根据错误类型生成不同响应。

## 3. 测试：单元测试与集成测试

**标注：核心高频**

Rust 内置测试框架，单元测试放在模块内，集成测试放在 `tests/` 目录。

```rust
pub fn normalize_name(input: &str) -> String {
    input.trim().to_lowercase()
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn normalize_name_should_trim_and_lowercase() {
        assert_eq!(normalize_name(" Alice "), "alice");
    }
}
```

**为什么这样设计、这样设计带来的好处：**

内置测试降低工程门槛，`cargo test` 成为所有项目通用入口，适合接入 CI。

## 4. 日志与追踪

**标注：重要**

生产服务推荐使用 `tracing`，它支持结构化日志和异步上下文追踪。

```toml
[dependencies]
tracing = "0.1"
tracing-subscriber = "0.3"
```

```rust
use tracing::{info, instrument};

#[instrument]
fn process_file(path: &str) {
    info!(path, "start processing");
}

fn main() {
    tracing_subscriber::fmt::init();
    process_file("app.log");
}
```

**为什么这样设计、这样设计带来的好处：**

普通日志只能记录文本，结构化追踪可以携带字段和调用跨度，便于排查复杂系统问题。

## 5. CI 与质量门禁

**标注：重要**

推荐最小质量门禁：格式化、静态检查、测试。

```yaml
name: Rust CI
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - run: cargo fmt --check
      - run: cargo clippy -- -D warnings
      - run: cargo test
```

**为什么这样设计、这样设计带来的好处：**

CI 把团队规范自动化，避免依赖人工记忆执行检查。

## 动手练习

### 练习 1：重构单文件项目

把一个 `main.rs` 项目拆成 `lib.rs`、`config.rs`、`service.rs`。

**思路提示：** 先移动纯逻辑，再让 `main.rs` 调用库函数。

### 练习 2：统一错误处理

用 `thiserror` 定义应用错误类型，替换 `unwrap`。

**思路提示：** 文件错误使用 `#[from]` 自动转换。

### 练习 3：补测试

为配置解析函数补单元测试和集成测试。

**思路提示：** 单元测试覆盖边界，集成测试覆盖整体流程。

### 练习 4：接入 tracing

给关键业务函数加 `#[instrument]`。

**思路提示：** 日志字段要包含能定位问题的业务 ID。
